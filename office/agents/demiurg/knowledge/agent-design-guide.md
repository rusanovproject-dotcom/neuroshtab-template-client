# Гайд по проектированию агентов (синтез Brahma + Cherry, март 2026)

## Когда что использовать

### Скилл (внутри агента)
- Разные стили/режимы одной работы (например, стили текста)
- Тот же контекст, другая инструкция
- "Один человек мог бы сделать без переключения контекста"
- Пример: Copywriter + скиллы write-post, write-ad, write-email, review

### Субагент (отдельный процесс)
- Независимая проверка (нужна изоляция от основного)
- Другая компетенция (текст + дизайн = два агента)
- Параллельная работа над разными частями
- Пример: написать текст (Copywriter) + проверить факты (Intelligence)

### Hook (автоматика)
- Детерминированные вещи: форматирование, линтинг, git safety
- PreToolUse = гейткипер (блокирует опасные операции)
- PostToolUse = валидатор (проверяет результат)
- Stop = финализатор (обновляет логи)

### MCP/Плагин (внешний инструмент)
- Доступ к внешним сервисам (Notion, Slack, Gmail)
- Базы данных, API
- Файловые операции за пределами workspace

## Архитектура идеального агента

### CLAUDE.md (100-200 строк макс)
1. Роль + конкретное поведение (не "помощник")
2. Контекст: handle-паттерн (пути к файлам, НЕ содержимое)
3. Протокол работы: 5 шагов
4. 2 примера Input → Output (бьют 10 правил)
5. Границы: "передай другому" с конкретными именами
6. Self-check чеклист перед отдачей
7. Output contract: что именно отдаёшь
8. Guardrails: стоп-краны

### Структура файлов
```
agents/{name}/
├── CLAUDE.md          ← ядро (100-200 строк)
├── skills/            ← режимы работы (по папкам)
│   ├── {skill-1}/SKILL.md
│   └── {skill-2}/SKILL.md
├── knowledge/         ← контекст (загружается по запросу)
│   ├── {domain}.md
│   └── examples/
└── state/
    └── context.md     ← текущий контекст агента
```

### Progressive Disclosure (3 уровня загрузки)
- L1: Metadata (name + description) — ВСЕГДА в контексте (~100 слов)
- L2: Body (SKILL.md) — когда скилл триггернулся (до 500 строк)
- L3: Resources (references/) — когда body говорит "прочитай X"

## Паттерны качества

### Self-Refine петля
Агент оценивает свой результат:
- Раунд 1: Сделай → Оцени → Найди слабое → Исправь
- Раунд 2: Пересчитай → Все >= 9? → ВЫХОД
- Стоп: avg >= 9, или delta < 0.5, или 3 раунда макс

### Pushy Descriptions (скиллы триггерятся)
Description должен быть навязчивым:
- "Use this skill when..." + 5 ситуаций
- Синонимы ключевых слов
- MANDATORY TRIGGERS секция

### Стиль написания
- Why > MUST (объясни причину, не приказывай)
- Негация → позитив ("переиспользуй" вместо "не создавай")
- 2 примера бьют 10 правил
- CLAUDE.md растёт только из ошибок (правило = Claude накосячил дважды)

## Командное взаимодействие (6 агентов)

### Dispatcher + Pipeline гибрид
Director — точка входа, маршрутизирует задачи.

### Structured Handoff (при передаче задач)
```
from: кто передаёт
to: кому
task: что делать
brief: тема + угол + формат
decision_context: ПОЧЕМУ именно это (агент не гадает)
output_expected: что ожидаем на выходе
```

### Quality Gate (3 уровня)
1. Self-check — внутри агента, чеклист перед отдачей (всегда)
2. Peer review — другой агент проверяет (для важных)
3. Approval — {{CLIENT_NAME}} подтверждает (для критичных: публикация, деплой, траты)

## BP: Best Practices (ключевые)
- BP-01: 2 узких агента лучше 1 широкого
- BP-02: Context Engineering > Model Selection
- BP-03: Explicit State Management (состояние явное, версионированное)
- BP-04: Observability First (логи, метрики, traces)
- Handle-паттерн: CLAUDE.md содержит ПУТЬ к файлу, не содержимое
- Одна сессия = одна задача (/clear между несвязными)

## Антипаттерны (не повторять)
- AP-01: Телефонный перевод — передавать задачу без decision_context
- AP-03: Дубли ответственности — два агента за одно и то же
- AP-14: 6+ агентов без явного Dispatcher = хаос
- Глава без подчинённых = бесполезная обёртка
- CLAUDE.md > 200 строк = перегружен
- Inline контент (содержимое файла скопировано в CLAUDE.md)
