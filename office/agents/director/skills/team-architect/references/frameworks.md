# Frameworks — Справочник фреймворков

## Span of Control

| Тип работы | Span | Источник |
|-----------|------|----------|
| Сложная/творческая (AI-агенты) | 3-5 | McKinsey |
| Стандартный менеджмент | 5-9 | Gallup |
| Пик вовлечённости менеджера | 8-9 | Gallup 2025 |
| Рутинная/стандартизированная | 10-15 | McKinsey |
| Netflix инженеры | 6-8 | Netflix |
| Agent Team (Claude Code) | 3-5 | Claude docs |
| AI-офис Director (текущий) | 7 | Факт |

**Правила для AI-офиса:**
- Director -> 6-9 агентов = норма
- Director -> 9+ = вводить Cluster Lead
- Каждый Cluster Lead -> не более 5-7 агентов
- Agent Team -> 3-5 teammates, 5-6 задач на teammate

**Оценка:**
- Span <= 5: возможен недогруз координатора
- Span 5-9: оптимум
- Span 10-12: зона риска, мониторить
- Span > 12: обязательная иерархия

## Формула Брукса

**C = n * (n-1) / 2** — каналы коммуникации

| n (агентов) | C (каналов) | Overhead |
|-------------|-------------|----------|
| 3 | 3 | Низкий |
| 5 | 10 | Низкий |
| 7 | 21 | Средний |
| 10 | 45 | Высокий |
| 12 | 66 | Высокий |
| 15 | 105 | Критичный |
| 20 | 190 | Критичный |

Рост команды вдвое = рост коммуникации в ~4x.

## RACI для AI-офиса

| Роль | Значение |
|------|----------|
| R — Responsible | Делает работу |
| A — Accountable | Approves результат (всегда 1) |
| C — Consulted | Даёт input до выполнения |
| I — Informed | Получает update после |

**Шаблон:**

| Задача | Director | Agent | {{CLIENT_NAME}} | Knowledge |
|--------|----------|-------|--------|-----------|
| Контент | C | R | A | I |
| Стратегия продукта | R | C | A | I |
| Деплой | C | R | A | - |
| Исследование | I | R | A | C |

**Правила:**
- Каждая задача: минимум 1 R, ровно 1 A
- Два A = нет A
- Много C = заторы. Резать.
- Много I = шум. Резать.

**DACI** (для решений, не задач):
- D (Driver) = двигает процесс -> Director
- A (Approver) = финальное слово -> {{CLIENT_NAME}}
- C (Consulted) = экспертиза -> агенты
- I (Informed) = узнают результат -> остальные

## Google Project Aristotle — Team Health Score

5 факторов (в порядке важности):

| # | Фактор | Что оценивать | Метрика для AI |
|---|--------|--------------|----------------|
| 1 | Психологическая безопасность | Агент эскалирует без страха? | Есть протокол эскалации, нет "наказания" за "не знаю" |
| 2 | Надёжность | Output соответствует контракту? | Output contract в CLAUDE.md, self-check |
| 3 | Структура и ясность | Роли чёткие? | CLAUDE.md с "отвечает/не отвечает", нет пересечений |
| 4 | Смысл работы | Агент знает зачем? | Связь с продуктом в knowledge/ |
| 5 | Влияние | Результаты видны? | ops/log.md обновляется, impact логируется |

**Шкала:** каждый фактор 1-5, итого 5-25.

| Score | Статус | Действие |
|-------|--------|----------|
| 20-25 | High-performing | Поддерживать |
| 15-19 | Нормально | Точечные улучшения |
| 10-14 | Проблемы | Разбирать по факторам |
| < 10 | Критично | Пересборка |

## Lean Waste — 7 типов для AI-команд

| Waste | Пример в AI-офисе | Решение |
|-------|-------------------|---------|
| Overproduction | Агент делает больше чем нужно | Output contract |
| Waiting | Агент ждёт context от другого | Handoff с полным контекстом |
| Motion | Ищет информацию в 3 местах | Единый источник правды (context.md) |
| Defects | Output не соответствует ожиданиям | Self-check перед отдачей |
| Over-processing | Лишние шаги в скилле | Аудит скиллов |
| Inventory | Незавершённые задачи в бэклоге | WIP limits |
| Transportation | Лишние передачи между агентами | Прямое выполнение |

## Theory of Constraints — 5 шагов

1. **Identify** — найди 1 главный bottleneck
2. **Exploit** — максимально загрузи bottleneck (не простаивает)
3. **Subordinate** — всё подстраивается под bottleneck
4. **Elevate** — увеличь пропускную способность (дублирование, автоматизация, декомпозиция)
5. **Repeat** — найди следующий bottleneck

**Принцип:** оптимизация не-bottleneck = waste. Только bottleneck улучшает систему.

**Как найти bottleneck в AI-офисе:**
- У кого самая длинная очередь задач?
- Кого чаще всего ждут другие агенты?
- Кто чаще всего является зависимостью в handoff?

**Решения для bottleneck:**
1. Дублирование — второй экземпляр агента
2. Декомпозиция — разделить задачи между несколькими
3. Автоматизация — дать агенту скиллы/MCP для ускорения
4. Переработка процесса — убрать зависимость от bottleneck

## Стадии роста команды

| Стадия | Агентов | Структура | Director |
|--------|---------|-----------|----------|
| Startup | 1-5 | Плоская | Player-coach |
| Growth | 5-10 | Роли с границами | Coordinator |
| Scale | 10-20 | Cluster Leads | Стратег |
| Enterprise | 20-50 | Отделы | Governance |

**Пороги перехода:**
- 5 агентов: тесная рабочая группа (Dunbar)
- 9 агентов: верхняя граница span -> готовить кластеры
- 12 агентов: обязательная иерархия
- 15 агентов: промежуточная иерархия (Dunbar)
- 50 агентов: departmentalization
