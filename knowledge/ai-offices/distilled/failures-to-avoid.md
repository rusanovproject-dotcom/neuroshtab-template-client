# Гнилые паттерны — что НЕ делать

> Дистиллят failure-mode'ов и анти-паттернов из аудита client-office-template, ревью Architect-of-Order и lessons-learned продакшена. Читать ПЕРЕД любой задачей где можешь сделать одну из этих ошибок.

Главная мысль: **большинство провалов AI-офиса не от плохой модели, а от структурных дыр.** Декларация без enforcement, дубли без dedup, hardcoded там где должно быть динамическое — типичные источники боли. Каждый паттерн ниже — реальный кейс из этого исследования.

---

## 1. Пустые failures.md — обучаемость нулевая

**Паттерн.** Каждый агент имеет `failures.md` по дизайну, но все они — пустые шаблоны с заглушкой «(пока пусто)».

**Почему гнилой.** Декларация в core.md *«grep по failures.md перед задачей»* работает в пустоту. Шаблон не учится. Через 3 месяца у клиента — те же ошибки что в день 1.

**Правило-замена.** Затравочно записать в `failures.md` каждого агента **1-2 реальных кейса** из истории lessons-learned (продакшена или подобных проектов). Так клиент увидит формат, а не пустую заглушку.

Пример из `research/03-internal-knowledge.md:122`:

```markdown
### 2026-04-04 — Notion MCP 401 token invalid
**Кто:** Director
**Что предположил:** локальный токен MCP подойдёт
**Что оказалось:** токен протух, нет ротации
**Правило:** только Cloud MCP с OAuth, локальный = fallback
```

**Источник.** `research/07-audit-client-template.md` P0-8: *«Все failures.md пустые — обучаемость нулевая. Декларация «grep перед задачей» работает в пустоту. Шаблон не учится.»*

## 2. Дублирование MCP — локальный + Cloud одновременно

**Паттерн.** Два MCP с одинаковой функцией активны одновременно (например, локальный Notion + Cloud Notion).

**Почему гнилой.** Путаница какой работает. Tools обоих видны Claude, он может выбрать неправильный. Когда один протухнет (локальный токен) — отчёты «не работает» хотя другой работает.

**Правило-замена.** Выбрать **один** канал. Рекомендация: Cloud MCP через OAuth (нет токенов в коде, нет ротации). Локальный — fallback для оффлайн-разработки.

**Источник.** `research/03-internal-knowledge.md:451`: *«Notion MCP имел две версии одновременно: локальная (с хардкоженным токеном, протухла) и Cloud (через OAuth, работает). Рекомендация: Вариант B — удалить локальный, использовать только Cloud.»*

## 3. Хардкод токенов в args MCP

**Паттерн.** API-токен прямо в `args` строке MCP конфига или в коде агента.

**Почему гнилой.** Токены протухают (срок жизни). При протухании — 401 ошибки, нет резервного канала. Опасно при коммите случайно.

**Правило-замена.** Использовать **env переменные** (`process.env.NOTION_API_KEY`) или **Cloud MCP через OAuth**. `.env` в `.gitignore`, в репо только `.env.example` без значений.

**Источник.** `research/03-internal-knowledge.md:450`. Lessons-learned 2026-04-04.

## 4. Self-validation без independent reviewer

**Паттерн.** Builder сам себя проверяет и выставляет score «у меня всё хорошо, 92/100».

**Почему гнилой.** Автор слеп. Self-review систематически завышает на ~20 пунктов. Реальный кейс: Demiurg self-score 92, реальный 74. Демиург смотрел изнутри пака, не на пересечение с шаблоном — пропустил 4 P0 блокера.

**Правило-замена.** Любой значимый артефакт — через **Builder/Validator** в РАЗНЫХ контекстах. Validator с read-only frontmatter (`tools: Read, Grep, Glob`). Если артефакт критичный — даже **новый чат** для ревьюера (чистый контекст).

**Источник.** `research/08-reviewer-report.md:475`: *«Demiurg систематически завышал на ~20 пунктов из-за слепых пятен self-review. Любой агент-инспектор должен сначала прогнать себя.»* См. `distilled/builder-validator.md`.

## 5. Install-agent скилл без поддержки новых ключей

**Паттерн.** В `install-agent/SKILL.md` не реализована поддержка новых ключей в `install.md` пакета (например, `archive_skills:`, `settings_json:`, `first_task:`).

**Почему гнилой.** Пак говорит «при установке заархивируй старый /audit-project и создай settings.json» — но install-agent игнорирует эти поля. Архитектура проектируется в одном месте, не работает в другом.

**Правило-замена.** При появлении нового ключа в install.md — **сразу обновить install-agent/SKILL.md**. Иначе функция декларирована, но не работает. Пользователь думает что сработало, на деле нет.

**Источник.** `research/FINAL-REPORT.md:144`: *«install-agent не понимает новые ключи archive_skills: и settings_json:. При установке Архитектора через /install-agent architect-of-order архивирование /audit-project и enforcement .claude/settings.json НЕ произойдут автоматически.»*

## 6. Trigger collision между скиллами

**Паттерн.** Два скилла подписаны на одну фразу. Например `/audit-project` и `/office-architect` оба слушают «проверь офис», «наведи порядок».

**Почему гнилой.** Director не знает кому передать. Скорее всего выберет первый по очереди. Второй скилл никогда не запустится. Иронично — Routing Validator (M3 сам по себе скилл) при первом запуске пометит COllison как P0 на сам себя.

**Правило-замена.**

1. **Anti-triggers обязательны** в frontmatter:

```yaml
TRIGGERS: "проверь офис", "наведи порядок"
ANTI-TRIGGERS: "аудит проекта" → /audit-project (в составе /build);
               пользователь работает над Tier 1 → не отвлекать
```

2. **При установке нового агента** install.md должен **переписать** существующие маппинги, не просто добавить рядом.

3. **Disambiguation step в install-agent** — если по ключевику нашлось 2+ пака — спроси у пользователя.

**Источник.** `research/08-reviewer-report.md:34`: *«Триггеры «проверь офис», «наведи порядок» УЖЕ занятые в шаблоне за `/audit-project`. install.md тупо добавит дублирующую строку. Routing Validator при первом же запуске выдаст trigger collision как P0 finding на сам себя.»*

## 7. Hardcoded маппинги к несуществующим пакам

**Паттерн.** В `install-agent/SKILL.md` жёстко прописан маппинг ключевиков:

```markdown
«копирайтер» → copywriter
«продажник» → sales
«tech lead» → tech-lead
```

— но `copywriter` пак без `install.md` (поставить нельзя), `sales` и `tech-lead` не существуют в `_agent-packs/`.

**Почему гнилой.** Клиент скажет «нужен копирайтер» — install-agent попытается поставить, не найдёт `install.md` — выдаст «битый манифест → стоп». Клиент думает «бот сломался», хотя в коде функция «есть».

**Правило-замена.** Маппинг **динамический** — читается из `_agent-packs/INDEX.md` каждый раз. Если пак не существует — честный fallback: «такого помощника пока нет, могу так через [агент]».

**Источник.** `research/07-audit-client-template.md:158`: *«В install-agent/SKILL.md:36-40 маппинг ключевиков жёстко прописан в коде: «копирайтер» → copywriter, «продажник» → sales, «tech lead» → tech-lead. Но copywriter не имеет install.md, sales и tech-lead не существуют.»*

## 8. Создавать новые папки без INDEX.md

**Паттерн.** Папка с 3+ файлами создана, но без `INDEX.md`.

**Почему гнилой.** Файлы теряются, поиск затруднён. Director не знает что в папке. Architect-of-Order при scan флагит как «orphan folder».

**Правило-замена.** Папка с **3+ файлами** обязана иметь `INDEX.md`. Это часть DEF-OF-DONE. Часть GOVERNANCE.md протокола (`research/03-internal-knowledge.md:31`).

**Источник.** `research/03-internal-knowledge.md:453`: *«Создавать новые папки без INDEX.md → файлы теряются, поиск затруднён, CLEANUP находит "призраков". Папка с 3+ файлами → ОБЯЗАН INDEX.md (part of DEF-OF-DONE).»*

## 9. Семантический баг центральной карточки

**Паттерн.** Файл назван `client-profile.md` и подразумевается как «профиль владельца офиса», но его шаблон — это карточка B2B-лида (LEAD_SOURCE, STATUS, Текущая сделка, Бюджет, Возражения).

**Почему гнилой.** Клиент при первом setup получит «Бюджет» и «Возражения» в свой собственный профиль. Сразу теряет доверие к шаблону. Это **первое** что видит клиент в первые 10 минут — провалить здесь = потерять клиента.

**Правило-замена.** `client-profile.md` в `office/` — это **профиль владельца** (имя, ниша, цели, активные проекты, стек, ритм работы). B2B-лиды живут в `clients/<slug>/client-profile.md`.

**Источник.** `research/07-audit-client-template.md:387`: *«Семантический баг office/client-profile.md — B2B-формат для самого пользователя. Клиент при первом setup получит «Бюджет» и «Возражения» в свой профиль. Сразу теряет доверие к шаблону.»*

## 10. Декларация защит без enforcement

**Паттерн.** install.md обещает «двойная защита `.env`» через settings.json deny-rules. Но `.claude/settings.json` в шаблоне **отсутствует**. Защита только prompt-уровневая (правило в core.md G4 «не читаю .env»).

**Почему гнилой.** Promise vs reality. Клиент откроет пак, увидит «Critical security recommendation: защита `.env`» — поверит. На самом деле — **одинарная защита**: если пользователь или Director скажут «прочитай .env чтобы найти hardcoded secrets», `Read` в allowed-tools РАЗРЕШИТ читать. Prompt G4 — единственный barrier.

**Правило-замена.** Либо install.md **создаёт** `.claude/settings.json` если его нет, либо **деградирует в installation failure**: «не ставлю пак пока в офисе нет .claude/settings.json». Третий вариант — физически переписать `allowed-tools` на узкий Read scope: `Read(office/**)`, `Read(.claude/**)`.

**Источник.** `research/08-reviewer-report.md:96`: *«Двойная защита .env — декларация без enforcement. Это критичный gap между «обещанием» и реальностью. Title секции в install.md «Critical security recommendation» — намекает что критично. Но рекомендация ≠ enforcement.»*

## 11. Пустые папки без README или .gitkeep

**Паттерн.** Папка `knowledge/scout/` пустая, в git коммитится только структура. Hardcoded в `build/SKILL.md:44` как место для досье `<домен>-YYYY-MM-DD.md`, но клиент не знает зачем эта папка.

**Почему гнилой.** Клиент откроет — ничего не понятно. Назначение нигде не объяснено. Если папка пустая — её удалят при следующей чистке как orphan.

**Правило-замена.** Каждая пустая папка — либо `.gitkeep` + 1-строчный `README.md` с назначением, либо удалить. Либо в шаблоне сразу 1-2 примера-затравки чтобы видна структура.

**Источник.** `research/07-audit-client-template.md:97`: *«knowledge/scout/ пустая, в git коммитится только структура... но клиент не знает зачем.»*

## 12. Дрейф между установленной версией и пакетом

**Паттерн.** `office/agents/designer/core.md` — 301 строка. `_agent-packs/designer/core.md` — 308 строк (свежее, включает скилл vercel-deploy). Если клиент сделает `/update-office`, неясно что подтянется.

**Почему гнилой.** Два варианта одного агента живут параллельно. Update-process не знает что выбрать. Клиент пишет «обнови дизайнера» — получает... непредсказуемый результат.

**Правило-замена.** Один источник правды:
- Либо `office/agents/designer/` — установлено, в `_agent-packs/designer/` не дублируется.
- Либо в `_agent-packs/designer/` — каноничная версия, при `/update-office` синхронизируется.

CI/pre-commit hook проверяет рассинхрон.

**Источник.** `research/07-audit-client-template.md:227`: *«Дрейф между office/agents/designer/core.md (301) и _agent-packs/designer/core.md (308). Свежий пак содержит секцию vercel-deploy и шаг про деплой. Установленный — нет. Если клиент сделает аудит — будет показано «Designer на старой версии».»*

## 13. Backup-папки в production-репо

**Паттерн.** `_agent-packs/designer-backup-2026-04-21/` — 268KB мёртвого кода. Заголовка нет, только дата в имени.

**Почему гнилой.** Backup в production-репо — антипаттерн. Раздувает репо, путает Architect-of-Order при scan (он флагит как orphan), занимает контекст когда грузится.

**Правило-замена.** Если нужна история — **git log** хранит её. Старые версии файлов — в `_archive/<YYYY-MM-DD>/` через append-only архивацию (не overwrite). Backup-папки — только в локальной разработке, не в коммите.

**Источник.** `research/07-audit-client-template.md:91`: *«Orphan-папка _agent-packs/designer-backup-2026-04-21/ — 268KB мёртвого кода. Backup в production-репо — антипаттерн.»*

## 14. Strategy.md существует ДО того как Strategist её собрал

**Паттерн.** В шаблоне `office/strategy/strategy.md`, `progress-index.md`, `roadmap.md` уже инициализированы плейсхолдерами `{{CLIENT_NAME}}` / `{{PROJECT_NAME}}`. То есть стратегия как бы существует до того как Strategist её собрал.

**Почему гнилой.** Если Director начнёт сессию — он прочитает `progress-index.md` (4264 строки скелета) и подумает «программа идёт». **Phase Lock сломан.** Семантика «стратегия пока не собрана» — нарушена.

**Правило-замена.** Стратегия **не предзаполнять**. Файлы создаются Strategist'ом из `office/agents/strategist/templates/` при первом запуске. До этого — папка `office/strategy/` пустая или с одним README.md «здесь будут strategy.md, roadmap.md, progress-index.md после Strategist'а».

**Источник.** `research/07-audit-client-template.md:368`: *«office/strategy/*.md уже частично заполнены плейсхолдерами и существуют до Strategist'а. Director прочитает «стратегия есть» когда её ещё нет. Phase Lock сломан.»*

## 15. Replace плейсхолдеров обещает 4, реальных 12+

**Паттерн.** `setup/SKILL.md` обещает заменить 4 плейсхолдера: `{{CLIENT_NAME}}`, `{{NICHE}}`, `{{NOTION_PROFILE_URL}}`, `{{GITHUB_USERNAME}}`. Реальных в шаблоне **больше**: ещё `{{PROJECT_NAME}}`, `{{DATE}}`, `{{DATE_W1}}`...`{{DATE_W6}}`, `{{LEAD_SOURCE}}`, `{{STATUS}}`, `{{CONTACT}}` и др. Все они **останутся** в шаблоне после setup и будут торчать как костыли.

**Почему гнилой.** Клиент после setup увидит `{{DATE}}` в strategy.md и подумает «бот сломался». Первое впечатление — провалено.

**Правило-замена.** В setup `Grep \{\{[A-Z_]*\}\}` по всему репо. Все найденные — заменить (явно указанное пользователем или дефолт «не указано» для опциональных). Никаких `{{...}}` после `/setup` — обязательная проверка в финале.

**Источник.** `research/07-audit-client-template.md:393` (P0-7): *«setup/SKILL.md обещает заменить 4 плейсхолдера, реальных 12+. Клиент после setup увидит {{DATE}} в strategy.md и подумает «бот сломался».»*

## 16. AI-слоп / hyperbole без citation

**Паттерн.** Тексты в скиллах/агентах содержат «инновационный», «синергия», «парадигма», «leverage», «exhaustive», «канон», «революционно». Восклицательные «Отлично!», «Замечательно!». Корпоративные обороты («таким образом», «в рамках данного»).

**Почему гнилой.** AI-фразы — маркер генерации, не обдуманного текста. Hyperbole без citation («устаканилось как канон») создаёт ощущение «промптом нагнал», не «реально обосновал».

Реальный пример из ревью (`research/08-reviewer-report.md:271`):

> *«README.md строка 14: «Реализует семейство ролей-«уборщиков», которое в индустрии в 2026 устаканилось как канон» — гипербола без citation. «Канон» — большое слово. Источники указаны, но они описывают как «Florian audit-prompt v5.0», «AutoDream» — это отдельные источники, а не «индустриальный канон». README претендует на больше чем есть.»*

**Правило-замена.** Каждое сильное утверждение — citation. Каждый бодрый эпитет — убрать. Перед коммитом — прогон через `stop-slop` skill (если есть) или ручная проверка по чек-листу:

- Нет «инновационный / синергия / парадигма / leverage / exhaustive»
- Нет восклицательных «Отлично!»
- Нет «таким образом», «в рамках данного»
- Если претензия большая — есть source-якорь

**Источник.** `research/08-reviewer-report.md:25-27` категория F «AI-слоп / наполнители» в ревью.

## 17. Дубли контента core ↔ soul ↔ knowledge ↔ SKILL

**Паттерн.** Те же таблицы повторяются в 3-х местах:
- `core.md:60-71` (таблица модулей)
- `knowledge/7-modules.md:13-19` (та же таблица)
- `core.md:97-108` (Phase Pipeline)
- `skills/<skill>/SKILL.md:91-198` (то же расширенно)

**Почему гнилой.** Layered design подразумевает «core = thin reference, knowledge = depth». Здесь core содержит **полные таблицы** которые потом ещё раз появляются в knowledge. Если все 5 файлов попадают в Claude session — клиент получает один и тот же материал 2-3 раза. Concept Pack читателя страдает от 30-40% избыточности.

**Правило-замена.** Один источник правды:
- **core.md** — workflow, references на knowledge.
- **knowledge/<topic>.md** — методология, depth.
- **SKILL.md** — workflow с триггером.
- **soul.md** — характер, биография, НЕ дублирует identity из core.

**Источник.** `research/08-reviewer-report.md:166`: *«core.md содержит полные таблицы которые потом ещё раз появляются в knowledge. Это раздувает контекст. Если все 5 файлов попадают в Claude session — клиент получает один и тот же материал 2-3 раза.»*

## 18. Soul.md повторяет identity из core

**Паттерн.** soul.md написан как extended core, не как самостоятельный документ.

Пример: core.md «Спокойный, точный, без иронии. Видишь каждую запятую» + soul.md «Спокойный. Точный. Без иронии. Без восклицаний.» — это reformulation, не биография.

**Почему гнилой.** Goal soul.md — характер, голос, душа. Не повтор identity из core. Если soul пустой — почему он есть?

**Правило-замена.** Soul.md — **биография** агента. Как у `_agent-packs/alex-marketer/soul.md` (96 строк): «3 круга жирной ЦА», «как работаю», «почему не лезу с советами», «3 книги в голове», чего не делаешь. Если нечего написать самостоятельного — soul.md не делать вообще (правило симметрии: либо у всех есть soul, либо ни у кого).

**Источник.** `research/08-reviewer-report.md:251`: *«Soul пустоват. Architect soul — как summary core.»*

## 19. False positives на специфике офиса (день 1 «zero learning»)

**Паттерн.** Architect-of-Order модуль M6 Failure Pattern Detector флагит «все failures.md пусты при работающем офисе >7 дней — обучаемость нулевая». Сценарий: клиент только что прошёл `/setup`, день 1, запускает «проверь офис» — получает P0 «zero learning» хотя офис работает 0 дней.

**Почему гнилой.** Generic-аудитор не знает специфики «свежий офис vs зрелый». Detector не учитывает дату создания репо или первого commit. Это самый частый сценарий — клиент только что взял шаблон, любопытен, запускает.

**Правило-замена.** Context-aware logic:

```
M6 Failure Detector =
  failures.md пусты
  И git log --reverse --format="%ai" | head -1 показывает что репо старше 14 дней
  И за последние 14 дней были коммиты от пользователя (не bot)
```

Без этого pattern detection даёт false positives в первые 2 недели жизни любого клиентского офиса.

**Источник.** `research/08-reviewer-report.md:197-215`.

## 20. Не запускать /audit-project / Architect dogfooding

**Паттерн.** Демиург собрал агента Architect-of-Order но **не запустил его на собственном паке** перед раздачей.

**Почему гнилой.** Один прогон поймал бы 2 из 4 P0 сразу: trigger collision сам с собой (P0-1) и heading collision «Module 7» (P0-4). Любой инспекционный агент может найти проблемы у себя — если ему дать шанс.

**Правило-замена.** Любой агент-инспектор сначала прогоняет себя. **Phase 9 Dogfooding** в `/build` для агентов которые могут проверить сами себя (Architect-of-Order, /audit-project, agent-quality-reviewer).

**Источник.** `research/08-reviewer-report.md:476`: *«Demiurg прошёл всё своё validation pipeline, но не сделал dogfooding. Один прогон поймал бы 2 из 4 P0 сразу. Любой агент-инспектор должен сначала прогнать себя.»*

## 21. Не пропускать Intake / Knowledge при сборке нового агента

**Паттерн.** Прыгнуть в BUILD минуя INTAKE и KNOWLEDGE-фазы Демиурга.

**Почему гнилой.** Без INTAKE — неясные требования = пустой агент, потом переделка. Без KNOWLEDGE — агент становится пустышкой, нет чем вооружить.

**Правило-замена.** В демиурге `/build`:
- INTAKE = блокер для Tier-1/2 (макс 2 раза вопросы, потом дается brief).
- Knowledge Mining ПЕРЕД Build, score < 5 = БЛОКЕР.
- Validate ОБЯЗАТЕЛЕН после Build, score >= 80 (PASS), 60-79 (max 2 iterate), <60 (FAIL).
- WIRE = обновить AGENTS.md + корневой CLAUDE.md (`@include`) + director/core.md (routing).

**Источник.** `research/03-internal-knowledge.md:455-458`. Lessons-learned демиурга.

## 22. Использовать `--dangerously-skip-permissions`

**Паттерн.** Запуск Claude Code с флагом `--dangerously-skip-permissions` чтобы не нажимать «approve» на каждый tool.

**Почему гнилой.** Реальный кейс 2026-04: Claude-Cursor агент удалил production-БД за 9 секунд. Источник: cxtoday.com, futurism.com, euronews.com — широко освещалось.

**Правило-замена.** Никогда вне изолированной dev-среды. Запретить через managed-settings (enterprise scope, перебить нельзя):

```json
{
  "permissions": {
    "deny": [
      "Bash(*--dangerously-skip-permissions*)"
    ]
  }
}
```

**Источник.** `research/04-governance-cleanup-agents.md:333`, `research/05-tools-services-mcp.md:291`.

## 23. Призраки в .env.example

**Паттерн.** В `.env.example` есть переменные которые никто не читает (например, `ANTHROPIC_API_KEY=` когда CLI не использует API key).

**Почему гнилой.** Клиент будет недоумевать зачем эта переменная. Возможно пойдёт получать ключ, потратит время. Когда поймёт что не нужна — потеряет доверие.

**Правило-замена.** Удалить из `.env.example` всё что **никто не использует**. Если переменная опциональная — пометить комментарием: *«если используешь API напрямую, не CLI»*.

**Источник.** `research/07-audit-client-template.md:61`: *«.env.example имеет ANTHROPIC_API_KEY= но ни один скилл его НЕ читает. Это призрак.»*

## 25. Хардкод GitHub URL без подписи / SHA-проверки

**Паттерн.** `install-agent/SKILL.md:50-51` использует hardcoded URL `github.com/<owner>/ai-office-packs`. Клиент при `/install-agent` пойдёт за чужим контентом без верификации.

**Почему гнилой.** Это и фича (общий магазин паков), и баг безопасности (клиент скачивает с чужого репо без подписи). Если злоумышленник получит доступ к репо паков — установит вредоносный код у всех клиентов.

**Правило-замена.** Подписать `_agent-packs/` через GitHub commit signing (GPG) или хотя бы добавить **SHA-проверку** при `install-agent` скачивании. install-agent проверяет hash скачанного против expected hash.

**Источник.** `research/07-audit-client-template.md:325`, P1-18 в табличке.

## 27. Knowledge запертая внутри пака агента

**Паттерн.** Knowledge JTBD заперта внутри пака `alex-marketer/knowledge/jtbd/`. Если клиент **НЕ** установит alex-marketer — JTBD-методологии у него нет. Если установит — копируется в `office/agents/alex-marketer/knowledge/jtbd/`, не в общую `knowledge/methodology/jtbd/`.

**Почему гнилой.** Knowledge принадлежит агенту, а не офису. Это означает что Strategist не сможет сослаться на JTBD-handbook (он в чужой папке).

**Правило-замена.** Общие методологии — в `knowledge/methodology/` верхнеуровневую. При установке пака — симлинк из `_agent-packs/<name>/knowledge/<topic>/` в `knowledge/methodology/<topic>/`. Доступно всем агентам.

**Источник.** `research/07-audit-client-template.md:260`: *«Knowledge JTBD заперта внутри пака alex-marketer. Knowledge принадлежит агенту, а не офису. Это означает что Strategist не сможет сослаться на JTBD-handbook.»*

## 28. Идти в Tier 4 пока есть открытые Tier 1

**Паттерн.** «Изучить новую тулзу», «оптимизировать тестовый покрытие», «настроить лучше CI» — пока есть открытые **Tier 1** задачи (клиент, КП, оффер).

**Почему гнилой.** Tier 4 = операционка / изучение / рефакторинг. Не приносит revenue. Жрёт время, которое могло пойти на закрытие Tier 1. Reverse Pareto: 80% эффекта от Tier 1, но всё внимание на Tier 4 потому что приятнее.

**Правило-замена.** Tier 4 ЗАПРЕЩЁН пока есть Tier 1. A+ Task = ВСЕГДА Tier 1 (если есть). Revenue ratio цель: ≥40% задач с label `revenue`.

**Источник.** Workspace CLAUDE.md, `research/03-internal-knowledge.md:331-345`. Это правило в каждом офисе должно быть в CLAUDE.md.

## 29. Pre-flight без content в файлах

**Паттерн.** В `core.md` агента написано «grep по failures.md перед задачей». Но `failures.md` пустой. Pre-flight выполняется формально, никаких уроков не извлекается.

**Почему гнилой.** Бесполезный ритуал. Декларация без content = ноль эффекта. Хуже того — агент думает «я выполнил pre-flight», уверенно идёт делать ту же ошибку второй раз.

**Правило-замена.** Pre-flight имеет смысл **только если** memory/failures наполнены реальным content. Перед раздачей шаблона — затравочно записать 1-2 реальных кейса в каждый `failures.md`. См. паттерн #1.

**Источник.** `research/03-internal-knowledge.md:124`: *«failures.md grep перед задачей обеспечивает не-повторение ошибок. НО — production ai-office-v2 НЕ использует failure.md систематически. Многие агенты (17 штук) не заполняют memory.md даже на проде.»*

## 30. Скиллы создавать с пустым description

**Паттерн.** SKILL.md с слабым `description:` — например, *«Скилл для маркетинга»*.

**Почему гнилой.** Anthropic: *«The skill's name and description fields are critical since Claude will use these when deciding whether to trigger the skill.»* Слабый description = скилл не вызывается. Висит мёртвым грузом.

**Правило-замена.** Description содержит:
- 1-2 предложения что делает.
- TRIGGERS секцию с конкретными фразами.
- ANTI-TRIGGERS секцию когда НЕ запускать.

**Discovery test:** спроси Claude *«какие скиллы есть для маркетинга?»* — должен назвать минимум 3 включая твой. Если не называет — переписать description.

**Источник.** `research/01-youtube-research.md:383`, `research/02-forums-communities.md:321`.

Перед коммитом любого изменения в офисе — пробежать этот список:

- [ ] Не заполнен ли я `{{...}}` плейсхолдер? Все ли заменены?

- [ ] Если добавил скилл — нет ли collision триггеров с существующими?
- [ ] Если правил установленный агент — обновил соответствующий пак (нет дрейфа)?
- [ ] Если изменил install.md ключи — обновил install-agent SKILL.md?
- [ ] Если декларирую защиту (deny-rules, anti-pattern) — есть ли enforcement?
- [ ] Если артефакт критичный (>50 строк, оффер, агент) — прошёл ли независимый review?
- [ ] Не использую ли AI-слоп («инновационный», «синергия», «парадигма»)?
- [ ] Не дублирую ли контент между core/soul/knowledge/SKILL?
