# /revise-segment — Утилита пятничной ревизии

> Это **единственная утилита** пайплайна распаковки ЦА. `/audience-stage` распаковывает сегмент с нуля, а эта утилита держит распакованное в актуальном состоянии — раз в неделю собирает накопленный `segment-observations.md`, переносит подтверждённое в `segment-portrait.md` и пересобирает `segment-core.md`.
>
> Без неё `segment-observations.md` превращается в свалку, `segment-portrait.md` устаревает, `segment-core.md` начинает врать. Поэтому утилита запускается **по расписанию**, а не «когда вспомнили».

---

📍 **Где это в системе:**
- **Уровень:** Maintenance utility (раз в неделю, пятница 16:00)
- **Запускается из:** расписания (cron / напоминание в календаре), триггеров клиента («пятничная ревизия», «обнови сегмент»)
- **Передаёт в:** обновлённые `segment-portrait.md` (v+1) + пересобранный `segment-core.md` + helmed `segments-map.md` (если возрождена убитая гипотеза)
- **Восстановление:** прерывание перед stop+wait на шаге 5 — diff остаётся в `_state/proposed-diff-{slug}.md`, можно продолжить с него

---

## 1. Цель

Пятничный ритуал 16:00 — переноса находок недели по цепочке `stream → dossier → base`.

Конкретно:
- Из `{slug}/segment-observations.md` за последние 7 дней вытащить подтверждённые находки (повтор у 2+ разных людей)
- Сформировать **proposed diff** к `{slug}/segment-portrait.md` — что добавить, что переформулировать, что удалить
- Получить апрув маркетолога/владельца (stop+wait)
- Применить diff, инкрементировать `version` dossier, дописать changelog в каждый изменённый блок
- Обновить `evidence_count` per unique person (новый респондент = +1, повтор того же человека = 0)
- Автоматически пересобрать `{slug}/segment-core.md` по правилам `_pipeline/derive-base-rules.md`
- Пометить обработанные stream-записи `→ merged to dossier v{N+1}` (НЕ удалять — append-only инвариант)

На выходе: dossier актуальный, base пересобран, stream размечен, валидаторы зелёные.

---

## 2. Когда запускать

**Автоматически — пятница 16:00.**

- При **1-3 активных сегментах** — global weekly: одна ревизия на все сегменты сразу за 30 мин.
- При **4-15 активных** — per-segment ротация раз в 2 недели (один сегмент за пятницу).
- При **15+** — see `OPERATIONS.md` (per-segment SOURCES, ротация).

Триггер cron — отдельный watcher OpenClaw (`audience/watcher`) шлёт маркетологу напоминание в TG в пятницу 16:00 со списком сегментов готовых к ревизии и команду `/revise-segment {slug}`.

**Вручную** — `/revise-segment {slug}` или `/revise-segment --all` (глобальная ревизия всех активных).

**НЕ запускать** если в stream сегмента 0 новых записей за неделю — нет смысла перетряхивать dossier ради нуля изменений. Утилита сама проверит и выйдет с сообщением «нечего ревизовать».

## 3. Вход

| Параметр | Тип | Описание |
|---|---|---|
| `{slug}` | строка | slug одного сегмента, например `h-osnovatel-agentstva` |
| `--all` | флаг | глобальная ревизия — пройтись по всем активным сегментам из NORTH-STAR.ТОП-3 |
| `--update-northstar` | флаг (опционально) | после ревизии запустить `/derive-northstar` (когда он появится — пока в Backlog) |
| `--dry-run` | флаг (опционально) | сформировать diff и показать, но не применять (для отладки) |

**Без параметров** — утилита спрашивает «какой сегмент ревизуем» и предлагает список из NORTH-STAR.

## 4. Девять шагов

Каждый шаг — атомарный. Если падает один — стоп, не идём дальше.

### Шаг 1 — Чтение stream за последнюю неделю

Открыть `audience/segments/{slug}/segment-observations.md`.

Отфильтровать записи **за последние 7 дней** по дате (формат записей в stream: `YYYY-MM-DD — #тег — цитата — источник`).

**Граница недели:** все записи с `date >= today() - 7`.

**Что игнорируем:**
- Записи уже помеченные `→ merged to dossier v{N}` — они уже в каноне.
- Записи помеченные `[игнор]` или `[ошибка]` — маркетолог забраковал руками.

**Если за неделю 0 новых записей** — выходим с сообщением «stream пуст, ревизия не нужна». Это не ошибка, это нормальный сценарий для тихой недели.

**Если 50+ записей за неделю** — флаг «слишком много, разбейся на 2 сессии». Обычно это сигнал что был залп custdev'ов и ревизию лучше делать частями.

### Шаг 2 — Группировка по тегам

Разложить отфильтрованные записи по тегам в **рабочий буфер** `_pipeline/revise-workspace-{slug}.md`:

| Тег | Что ищем |
|---|---|
| `#боль` | формулировки «не могу», «теряю», «болит» |
| `#возражение` | «дорого», «не сейчас», «у меня не получится» |
| `#страх` | «боюсь что», «вдруг» |
| `#триггер` | конкретное событие — «после потери клиента», «когда выгорел» |
| `#цитата` | дословная фраза для lingvo |
| `#конкурент` | упоминание чужого продукта |
| `#инсайт` | непонятно куда, но важно — разбираем на ревизии |

Внутри каждого тега — по подсегментам если есть (например, в #боль — «операционная нагрузка», «отсутствие системы», «страх делегирования»).

### Шаг 3 — Поиск подтверждений

**Правило:** запись становится **кандидатом в dossier** если повторяется у **2+ разных людей** в stream (с учётом всей истории stream, не только последней недели).

**Как считаем:**
- Берём кандидата (например, новую боль «не успеваю отвечать клиентам»)
- Ищем по всему stream близкие по смыслу записи (LLM-поиск через семантическое совпадение, не только grep)
- Считаем **уникальных людей** (имя респондента или slug источника в SOURCES.md)
- Если 2+ уникальных — кандидат в dossier
- Если только 1 — оставляем в stream, ждём третьего подтверждения

**Дополнительные кандидаты:**
- Записи которые **противоречат** существующему в dossier — кандидат на переформулировку
- Боли/возражения из dossier которые **не всплывали 2+ месяца** — кандидат на удаление

### Шаг 4 — Формирование proposed diff

Собрать diff в формате методологии (`OPERATIONS.md`, раздел «Ритуал ревизии»):

```
Сегмент h-slug — proposed diff (revise 2026-04-25):

+ В "Боли" добавить: "Не успеваю отвечать клиентам — теряю горячие сделки" (3 подтверждения от Анны М., Сергея B., Ольги-Т., evidence_count 3)
+ В "Возражения / ценовые" добавить: "Дорого без гарантии что окупится за месяц" (2 подтверждения)
~ Переформулировать возражение #2: было "не сейчас, дел много" → стало "не сейчас, не успею внедрить" (новая цитата от Виктора Ю. точнее)
~ Боль #4 "тону в задачах": evidence_count 4 → 5 (новый респондент Ольга-Т.)
- Удалить страх #4 "вдруг команда не примет" — не всплывал 2 месяца, последний раз 2026-02-18

Lingvo:
+ Добавить цитату "я как стрелочник на станции — все ко мне с мелочами" (Анна М., 2026-04-22)
```

Diff включает три типа правок:
- `+` — добавить новое
- `~` — переформулировать существующее (со ссылкой на новую цитату)
- `-` — удалить устаревшее (с указанием срока «не всплывал N месяцев»)

**Важно:** diff формируется **не на segment-core.md**, а на `segment-portrait.md`. Base пересобирается автоматически на Шаге 8.

### Шаг 5 — Stop+wait, апрув маркетолога/владельца

**ОБЯЗАТЕЛЬНЫЙ чек-поинт.** Ничего не пишем в файлы пока не получили апрув.

Утилита показывает diff в чате и ждёт ответ:

| Ответ | Действие |
|---|---|
| `да` / `apply` / `+` | применяем весь diff |
| `нет` / `stop` / `-` | откат, ничего не пишем, выход |
| `докрути по пунктам: 1, 3` | применяем только пункты 1 и 3, остальные в stream остаются |
| `переформулируй пункт N` | возвращаемся на Шаг 4, корректируем формулировку |

**Без этого шага утилита не идёт дальше.** Это инвариант — dossier правится только с апрува человека.

**Защита от автозапуска без человека:** если утилита запущена в неинтерактивном режиме (cron, watcher), она формирует diff, кладёт в `_pipeline/revise-workspace-{slug}.md`, шлёт TG-уведомление маркетологу с командой `/revise-segment {slug} --resume` и **выходит**. Применение — только при `--resume` с живым человеком.

### Шаг 6 — Применение diff к dossier с per-block changelog

После апрува:
1. Открыть `{slug}/segment-portrait.md`
2. Применить каждый пункт diff в соответствующую секцию (1.2 Боли, 1.4 Страхи, 1.5 Возражения, 4.1 Lingvo и так далее)
3. **В конце каждой изменённой секции** дописать строку в локальный changelog:
   ```
   <!-- changelog: v3 (2026-04-25) — добавлена боль #8 из stream-2026-04-22 (Анна М.), evidence_count 3 -->
   ```
4. В шапке dossier — `version: 3 → version: 4`
5. В нижнем глобальном Changelog — суммарная запись:
   ```
   - **v4** (2026-04-25): ревизия за неделю 2026-04-18 → 2026-04-25. +1 боль, +1 возражение, ~1 переформулировка, -1 страх. Источники — stream-записи 2026-04-22 (Анна М.), 2026-04-23 (Сергей Б.), 2026-04-24 (Ольга-Т.).
   ```

**Архивация старого dossier (тест T5.1):** перед перезаписью копия предыдущей версии уезжает в `_archive/{YYYY-MM-DD}_{slug}/segment-portrait.md` для истории и возможного отката.

### Шаг 7 — Инкремент evidence_count per unique person

**Правило О1 из плана:** evidence_count считается **per unique person**, не per mention.

| Сценарий | Действие |
|---|---|
| Новый респондент подтвердил боль | evidence_count `+1` |
| Тот же респондент повторил боль на 3 созвонах | evidence_count `+0` (это всё ещё 1 человек) |
| Боль появилась в stream впервые от 1 человека | evidence_count = 1, но в dossier пока **не идёт** (нужно 2+) |
| Маркетолог вручную перепроверил dossier и поправил цифры | evidence_count **сбрасывается** на ту цифру что выставил человек |

Источник правды по «уникальности человека» — поле `Имя` или slug в SOURCES.md. Если один и тот же человек проходит под разными именами (созвон + диагностика) — маркетолог склеивает на ревизии.

**Синтетика:** если evidence_count построен из синтетических интервью — пометка `[синт]` (например, `evidence_count: 4 [синт]` или `evidence_count: 4 (3 живых + 1 [синт])`). Гейт расхождения «синт vs живые ≤ 30%» — отдельная история, проверяется в Скилле №2, на ревизии не трогаем.

### Шаг 8 — Автоматическая пересборка segment-core.md

**Это та же логика что в Шаге 7 Скилла №3** — мы не выделяем `/derive-base` в отдельный скилл, переиспользуем код.

Запускаем правила `_pipeline/derive-base-rules.md` для всех 10 секций base:
1. Паспорт — мода по двум представителям
2. Боли (топ-7) — сортировка по `частота × LTV-импакт`, фильтр `priority: high`
3. Потребности (топ-7) — обратная сторона топ-7 болей
4. Страхи (топ-5) — объединённые «Ты + Я», по частоте
5. Возражения (топ-5) — балансный отбор 2+2+1 (ценовых/ценностных/доверительных)
6. Dream Outcome — на голосе клиента из самого яркого lingvo
7. Lingvo (топ 3-5) — частотность × выразительность
8. Триггеры (2-3) — только событийные, не абстрактные
9. Связка с продуктом — 1:1 mirror dossier.7
10. Anti-сегмент — одна строка из dossier.8.1

**После пересборки — quality gates:**
- [ ] Цитаты в base — подмножество `segment-portrait.md#lingvo`?
- [ ] Секция 9 (Связка) идентична dossier 1:1 (drift-check)?
- [ ] В секции 2 (Боли) у каждого пункта есть вес?
- [ ] Триггеры в секции 8 — конкретные события (не «когда дозреет»)?
- [ ] segment-core.md ≤ 150 строк (hard cap)?

Если хоть одно «нет» — **откат**, чиним dossier, заново derive. На production-окружении это блокер для коммита.

В шапке base ставим `derived_from: segment-portrait.md@v{N+1}` и `last_generated: YYYY-MM-DD`.

### Шаг 9 — Пометки в stream

Для каждой обработанной записи в `{slug}/segment-observations.md` дописываем суффикс:

```
2026-04-22 — #боль — "я как стрелочник на станции" — Анна М., созвон → merged to dossier v4
```

**НЕ удаляем** — stream остаётся append-only, история подтверждений сохраняется. На квартальной архивации (раз в 3 мес) merged-записи уезжают в `_stream-archive/YYYY-Q{N}.md`, в живом stream остаются pending + свежие merged за последний квартал.

## 5. Возрождение убитых гипотез (О3)

**Сценарий:** в `segments-map.md` есть гипотеза в секции «Убитые гипотезы» (например, «Студенты-айтишники — не подтвердилась, причина: низкая платёжеспособность»). Через 3+ месяца в stream появляются записи воссоздающие эту гипотезу — другой контекст, новая платёжеспособность, обновлённые данные.

**Правило:** старую запись **не редактируем** (история сохраняется), создаём **новую запись** в segments-map.md с явной ссылкой на убитую:

```
### Гипотеза #15 — Студенты-айтишники (возрождение)
- **Дата:** 2026-07-25
- **Статус:** 🟡 1-2 подтверждения
- **Откуда пришла:** возрождена 2026-07-25 по новому источнику (custdev с senior-студентами bootcamp), см. убитую гипотезу «Студенты-айтишники» от 2026-03-15
- **Что изменилось:** теперь смотрим на bootcamp-выпускников 25-32 с зарплатой 200К+, не на классических студентов
- **Подтверждения:**
  - {YYYY-MM-DD} — custdev, {профиль респондента}, готов платить {сумма}
```

**Как утилита это ловит:**
- На Шаге 2 при разборе #инсайт-записей сравнивает их с секцией «Убитые гипотезы» segments-map.md (LLM-семантика, не только slug-grep)
- Если совпадение ≥70% — флаг «возможное возрождение», добавляет в proposed diff отдельным пунктом
- Маркетолог подтверждает на Шаге 5 — утилита создаёт новую запись segments-map.md, не трогает убитую

## 6. Архив synthetic-interviews (О2)

**Сценарий:** маркетолог за неделю запустил повторный custdev (например, скорректировал OCEAN-персону и перегенерил интервью). В `{slug}/synthetic-interviews.md` лежит свежая версия, но предыдущая нужна как история (для аудита расхождений «синт vs живые» и для отката).

**Правило:** перед перезаписью `synthetic-interviews.md` текущая версия копируется в:

```
{slug}/_synthetic-archive/YYYY-MM-DD-vN.md
```

Где `N` — порядковый номер прогона (`v1` — первый custdev, `v2` — второй и так далее).

**Утилита ловит это так:**
- На Шаге 1 проверяет дату `synthetic-interviews.md` — если она моложе предыдущей merged-записи в SOURCES.md и пометка `тип: custdev-resync` — это сигнал что был повторный прогон
- На Шаге 6 (после апрува) копирует старую версию в архив, обновляет SOURCES.md строкой `merged → synth-archive v{N}`
- В живом файле остаётся последняя версия

Архивные файлы — read-only, не редактируем. На квартальной архивации не трогаем (это уже архив).

## 7. Acceptance criteria (план 8.1)

После успешного прогона `/revise-segment {slug}`:

- [ ] **segment-portrait.md обновлён**, в шапке `version` инкрементнут на 1 (T5.2)
- [ ] **per-block changelog** в каждом изменённом разделе (T5.3)
- [ ] **глобальный Changelog** dossier дополнен суммарной записью с датой и причиной
- [ ] **evidence_count** per unique person корректно обновлён (новый респондент = +1, повтор = 0)
- [ ] **segment-core.md пересобран**, в шапке `derived_from: segment-portrait.md@v{N+1}`, `last_generated` — сегодня
- [ ] **drift-check пройден** — секция 9 base = 1:1 dossier.7, цитаты подмножество lingvo, ≤150 строк, 0 плейсхолдеров `{}`
- [ ] **stream-записи помечены** `→ merged to dossier v{N+1}` (не удалены)
- [ ] **архив старого dossier** в `_archive/{YYYY-MM-DD}_{slug}/segment-portrait.md` (T5.1)
- [ ] **валидаторы зелёные** — `validate-dossier.sh`, `validate-base.sh`, `validate-citations.sh` exit code 0
- [ ] (если применимо) **возрождённые гипотезы** добавлены в segments-map.md новыми записями со ссылкой на убитые
- [ ] (если применимо) **synthetic-interviews** заархивированы в `_synthetic-archive/`

Если хоть один пункт «нет» — ревизия не считается завершённой. Маркетолог чинит и заново.

**Машинный валидатор** `validate-revise.sh` (если установлен в проекте) запускается автоматически в конце Шага 9.

## 8. Квартальная архивация (раз в 3 месяца, 15 минут)

Это **отдельный ритуал**, не часть `/revise-segment`, но утилита его триггерит флагом `--quartal-archive` (или вручную раз в 3 мес).

**Что делаем:**
- **Stream:** записи старше 3 мес или с пометкой `merged` → в `{slug}/_stream-archive/YYYY-Q{N}.md`. В живом stream остаётся только pending + свежие merged.
- **SOURCES:** строки со статусом `merged` или `rejected` старше 3 мес → `_sources-archive/YYYY-Q{N}.md`. В основном файле — pending + свежие.
- **Dossier:** **не архивируем** — это живой канон. Растёт через замещение, не добавление (новая боль вытесняет неподтверждённую старую).

Подробности — в `OPERATIONS.md` раздел «Квартальная архивация».

## 9. /derive-northstar (раз в месяц — упоминание)

**Что делает:** агрегирует Часть B `NORTH-STAR.md` (общие боли + Golden Quote) из активных сегментов по правилам `_pipeline/north-star-update.md`.

## 10. Связь с другими скиллами

| Скилл | Что общего |
|---|---|
| **Скилл №3 `/segments-awareness`** Шаг 7 | **Та же derive-base-rules логика.** Скилл №3 собирает base первый раз, эта утилита пересобирает его на ревизиях. Один и тот же код, одни и те же quality gates. Если меняется логика derive — меняется в обоих местах синхронно (это инвариант плана 8.1) |

| **Скилл №1 `/segments-discover`** | Поставляет segments-map.md с убитыми гипотезами — утилита проверяет их на возрождение |

## 12. Что внутри папки скилла

```
revise-segment/
├── SKILL.md                    (этот файл — методология и контракт)
├── validate-revise.sh          (машинный валидатор всех 9 шагов, exit 0/1)
└── examples/
    └── h-example-revision.md   (пример выполненной ревизии — diff, changelog, before/after)
```

Валидатор `validate-revise.sh` проверяет:
- Шаг 6 — version инкрементнут (T5.2)
- Шаг 6 — per-block changelog есть в каждом изменённом блоке (T5.3)
- Шаг 6 — старая версия dossier в `_archive/{date}_{slug}/` (T5.1)
- Шаг 8 — segment-core.md пересобран, drift-check ок, цитаты подмножество lingvo, ≤150 строк
- Шаг 9 — обработанные stream-записи помечены merged

Failure валидатора = ревизия не закрыта, маркетолог чинит и заново.

## 13. Краткий чек-лист (для оперативки)

Когда маркетолог в пятницу 16:00 видит TG-напоминание:

1. `/revise-segment {slug}` (или `--all` если 1-3 сегмента)
2. Прочитать stream за неделю (Шаг 1) — утилита покажет
3. Проверить группировку по тегам (Шаг 2) — утилита разложит
4. Прочитать proposed diff (Шаги 3-4)
5. **Апрувнуть / докрутить / отклонить** (Шаг 5 stop+wait)
6. Утилита сама применяет (Шаг 6), считает evidence (Шаг 7), пересобирает base (Шаг 8), помечает stream (Шаг 9)
7. Проверить что валидаторы зелёные
8. Готово — следующая ревизия через неделю

**Длительность всего цикла на 1 сегмент** — 10-20 минут чистого времени маркетолога. Остальное делает утилита.
