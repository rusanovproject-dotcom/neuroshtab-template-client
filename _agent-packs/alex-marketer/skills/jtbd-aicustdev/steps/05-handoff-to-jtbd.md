# Шаг 05. Handoff обратно в /jtbd Step 01b

**⛔ ПРАВИЛО КОММУНИКАЦИИ: одно короткое сообщение клиенту с переходом + пометка о статусе. Затем сразу запуск Step 01b основного /jtbd.**

## Цель

Вернуть управление в основной `/jtbd` со всеми правильно проставленными флагами:
- `synthetic-factbase-<main>.md` готов и помечен 🔴
- В шапке `JTBD_анализ_<main>.md` будет добавлен баннер `STATUS: synthetic-draft`
- В `agent-state.md` зафиксирован `factbase_source: synthetic` (или `hybrid`)
- `/jtbd Step 01b` (cluster-segments) запускается с источником = synthetic-factbase, не реальная фактура

---

## Алгоритм (молча)

### 1. Финальная сверка synthetic-factbase

Открой `synthetic-factbase-<main>.md` через `view`. Проверь по чек-листу:

- [ ] Шапка с 🔴 предупреждением на месте
- [ ] Секция «Параметры генерации» заполнена
- [ ] Секция «Кластеры цитат» содержит ≥ 1 кластера в каждой из 6 категорий (или явная пометка «не нашлось» если по какой-то категории не сформировалось)
- [ ] «Полные интервью» содержат все N персон, у каждой все 8 блоков
- [ ] Сводка для `/jtbd` в конце на месте
- [ ] Все цитаты помечены 🔴

Если что-то не так — не запускай handoff, вернись на нужный шаг и доделай.

### 2. Подготовка `JTBD_анализ_<main>.md` к работе на синтетике

Если файл `JTBD_анализ_<main>.md` ещё не создан (его создаст основной `/jtbd` Step 00) — этот пункт пропускается, баннер добавит сам `/jtbd` Step 00 когда увидит флаг `factbase_source: synthetic` в agent-state.

Если уже создан — добавь в самое начало (после `# JTBD-анализ: ...`) баннер:

```markdown
> ## 🔴 STATUS: synthetic-draft
>
> Документ построен на **синтетической фактуре** (AI-сгенерированные касдевы). Все цитаты в этом анализе помечены 🔴 AI-генерация. Это **временный черновик** для построения первой версии анализа.
>
> **Stage 1 НЕ закрывается** до сбора минимум **3-5 интервью с живыми платящими клиентами** и пересборки сегмента-фокуса на реальной фактуре. До замены — документ **не передаётся в Stage 2** (Product / Offer).
>
> Источник синтетики: `audience/synthetic-factbase-<main>.md` от [дата].
```

### 3. Обновление agent-state.md

```yaml
active_skill: /jtbd          # возвращаемся в основной скилл
active_step: 01b-cluster-segments-from-synthetic
factbase_source: synthetic   # или hybrid
synthetic_factbase_file: projects/<main>/audience/synthetic-factbase-<main>.md
synthetic_personas_count: N
last_checkpoint: YYYY-MM-DD HH:MM
synthetic_done_at: YYYY-MM-DD HH:MM
synthetic_replacement_due: «при первой возможности — минимум 3-5 живых интервью по сегменту-фокусу до закрытия Stage 1»
```

### 4. Обновление `JTBD_прогресс_<main>.md` (если используется)

Если `/jtbd` создал файл прогресса в проекте — допиши в него блок:

```markdown
- [completed] Сбор фактуры (через синтетику — синтетические касдевы 🔴)
- [in_progress] Группировка клиентов
```

## Что СКАЗАТЬ клиенту (одно сообщение)

> «Фактура собрана. Возвращаюсь в основной анализ — сейчас по этим данным сгруппирую клиентов в сегменты, потом по каждому пройдусь полным циклом (что у них главное, как описать, путь, конкуренты, барьеры, оценка).
>
> Помни — всё что соберём дальше будет с пометкой «черновик на синтетике». Перед тем как двигать в продукт / оффер, нужно эти портреты подтвердить на 3-5 живых клиентах. Я тебе напомню когда дойдём до закрытия.»

---

## Что в результате уходит в основной /jtbd

Алекс вызывает `/jtbd Step 01b (cluster-segments)` с явным флагом:

```
input_factbase: projects/<main>/audience/synthetic-factbase-<main>.md
mode: synthetic   # или hybrid
mark_all_quotes: 🔴 AI-генерация
```

Step 01b использует кластеры из `synthetic-factbase` как основу для группировки сегментов. Дальше — обычный цикл `/jtbd` по 13 шагам, **но во всей сводной таблице достоверности** все блоки сегментов получают пометку:
- 🔴 если режим = synthetic
- 🟡 если режим = hybrid (часть из реальных данных)
- 🟢 запрещено выставлять до замены синтетики на реальные интервью

---

## Триггер для /jtbd-critic

Когда основной `/jtbd` дойдёт до финала и предложит запустить `/jtbd-critic` — критик автоматически выдаст ⚠️ АУДИТ-блок про необходимость замены синтетики (см. правки в `/jtbd-critic/steps/01-flags.md` — флаг 4 расширен).

---

## Когда модуль закрывается

После handoff — `/jtbd-aicustdev` **не возвращается в активные**. В `agent-state.md` `active_skill` перешёл на `/jtbd`, дальнейшая работа идёт по основному скиллу.

Возврат к `/jtbd-aicustdev` возможен **только** если:
- Клиент явно попросил досгенерировать ещё N персон (нужно расширить выборку — например, добавить тип клиентов которого не было)
- При смене ниши / продукта (по сути это новый проект, и модуль запускается заново с Step 00)

---

## ЗАПРЕТЫ

- НЕ закрывай Stage 1 на этом шаге — это критическая ошибка. Stage 1 закрывается только после `/jtbd-critic` + замены синтетики на ≥3-5 живых интервью + явного «принимаю» от клиента
- НЕ снимай 🔴 пометку при handoff
- НЕ передавай в `/jtbd Step 01b` без флага `mode: synthetic` — без него /jtbd будет считать что фактура реальная
- НЕ забудь про баннер `STATUS: synthetic-draft` в `JTBD_анализ_<main>.md`
