# Audience Stage — главный скилл Stage 1

## Зачем

Клиент пришёл с базовым проектом от Стратега, в `projects/<main>/` уже есть наработки по ЦА (diagnostic-card, audience/, заметки) — общие, без сегментов, без awareness, без живых цитат. Этого мало для оффера.

Этот скилл создаёт **отдельную папку** `projects/<main>-audience/` под глубокую распаковку и проводит клиента через полный audience pipeline. На выходе — фундамент: КТО реально клиент, к кому двигаться, какие сегменты в приоритете.

После Stage 1 артефакты остаются в этой папке, могут использоваться напрямую или быть скопированы в основной проект.

## Когда запускать

**Явно (триггеры):**
- «распакуй мою ЦА», «найди настоящих клиентов», «давай audience», «stage 1»
- «распакуй ЦА с нуля», «начнём с целевой», «глубокая распаковка ЦА»
- `/audience-stage`

**Неявно — Алекс предлагает сам когда:**
- Клиент только что прошёл онбординг со Стратегом и спрашивает «что дальше?»
- В `projects/<main>/` есть `diagnostic-card.md` или `audience/` но нет `audience/segments/`
- Клиент жалуется на низкую конверсию или непонимание ЦА

**НЕ запускать:**
- Если уже есть `projects/<main>-audience/` с распакованными ТОП-3 → предложить `/revise-segment` или работу с конкретным сегментом
- Если клиент явно сказал «не распаковывай ЦА» / «у меня уже всё ок»

## Pre-flight checks

```
0. ⚠️ RESUME CHECK — обязательно ПЕРВЫМ шагом:
   if [ -f "projects/<main>-audience/_state/progress.md" ]; then
     # Сессия в процессе — НЕ запускать заново
     cat projects/<main>-audience/_state/progress.md
     Сказать клиенту:
       «Stage 1 уже в процессе. Текущая сессия: {из state.md → Current session}.
        Last completed: {из state.md → Last completed}.
        Next action: {из state.md → Next action}.

        (a) Продолжить с этого места — Stop+wait
        (b) Начать заново (старая папка → _archive/) — Stop+wait
        (c) Завершить разбор досрочно — отметить DONE в state.md»
     Stop+wait, без явного выбора не идти.
   fi
   # Если state.md нет — свежий старт, идём дальше по чеклисту.

1. Определить основной проект:
   - Один в projects/ кроме _template / _template-audience / _example-* → берём как <main>
   - Несколько → спросить «по какому распаковываем ЦА?»
   - Только шаблоны → «Создай проект через Стратега, потом вернёмся»

2. Проверить шаблон _template-audience:
   ls projects/_template-audience/audience/INDEX.md
   - Если нет → ошибка офиса, эскалация владельцу
     «Шаблон _template-audience отсутствует. Не могу продолжить.»

3. ⚠️ ПРОВЕРКА существующей папки <main>-audience/:
   if [ -d "projects/<main>-audience" ]; then
     • Сказать клиенту: «Папка projects/<main>-audience/ уже существует.»
     • Прочитать `projects/<main>-audience/CHANGELOG.md` — что там сделано
     • Предложить 3 варианта:
       (a) ПРОДОЛЖИТЬ — узнать на каком этапе остановились, идти дальше
       (b) НАЧАТЬ ЗАНОВО в `<main>-audience-v2/` — старая остаётся как архив
       (c) ОТМЕНИТЬ — остановиться
     • Stop+wait, без явного выбора не идти

4. Найти diagnostic-card:
   DIAG_PATH=""
   if [ -f "projects/<main>/audience/diagnostic-card.md" ]; then
     DIAG_PATH="projects/<main>/audience/diagnostic-card.md"
   elif [ -f "projects/<main>/diagnostic-card.md" ]; then
     DIAG_PATH="projects/<main>/diagnostic-card.md"
   fi
   - Если найден — читаем как starter (Шаг 2)
   - Если НЕ найден — короткий intake (Шаг 2 fallback)

5. Найти существующие audience-наработки Стратега:
   if [ -d "projects/<main>/audience" ]; then
     ls projects/<main>/audience/*.md
     # Читаем все .md как референс (НЕ перезаписываем!)
     # Возможные файлы: pains.md, segments.md, raw-segments.md, language.md
     # Все они идут как input для /segments-discover Этап 1
   fi

6. ⚠️ MODE DETECTION — определить режим работы:

   Алекс читает доступные данные и определяет один из 3 режимов:

   ESTABLISHED — есть продажи + ≥3 клиента + diagnostic-card от Стратега
     → Полный путь /audience-stage без модификаций
     → 9 вопросов в Этапах 0-4 опираются на реальный опыт клиента
     → segment-money-map подтверждается клиентами
     → Acceptance: NORTH-STAR содержит реальные сегменты с клиентскими цитатами

   EARLY-STAGE — есть продукт + 1-2 клиента ИЛИ есть продукт без продаж + есть личный опыт ниши
     → Полный путь, но с упором на синтетику в /segments-unpack (custdev)
     → segment-money-map даёт 70% картины (рынок), 30% от клиентов
     → Acceptance: NORTH-STAR содержит сегменты, помеченные «требует валидации в первых продажах»

   GREENFIELD — 0 продаж + 0 клиентов + только идея/ниша/желание
     → АДАПТАЦИЯ:
       • Шаг 4a вопросы 4-5 (про кейсы клиента) **переформулировать**:
         НЕ «Кому ты УЖЕ дал больше всего пользы?»
         А: «К кому ты сам обращался когда сталкивался с этой проблемой? И что у тебя есть из связанного опыта — личного / профессионального — что даёт тебе угол захода в эту нишу?»
       • Money-map усиленный — без клиентов всё опирается на рынок и аналоги конкурентов
       • segment-hypotheses через **обязательный custdev** (синтетические интервью OCEAN-персон) — без живых данных это единственный путь
       • Ось 7 scoring (соответствие ресурсам) опирается на личный опыт клиента и связи, не на кейсы (т.к. кейсов 0)
       • Pre-flight gate `/unpack-product` — будет требовать валидации NORTH-STAR через первые 5-10 продаж до перехода в Stage 2 продукт
     → Acceptance: NORTH-STAR с явной пометкой «🌱 GREENFIELD — гипотетические сегменты, требуют живой валидации в продажах перед /unpack-product»

   Алекс **спрашивает явно** перед стартом:

   > *«Чтобы выбрать правильный режим работы — у тебя сейчас:*
   >
   > *1. Есть продажи и хотя бы 3 клиента + работа со Стратегом → established (полный путь)*
   > *2. Есть продукт + 1-2 клиента или личный опыт ниши без клиентов → early-stage (с custdev)*
   > *3. Только идея, 0 продаж, начинаешь с нуля → greenfield (адаптированный путь)*
   >
   > *Какой ближе?»*

   Stop+wait, без явного выбора не идти.

   Записать в `_state/pipeline-progress.md`:
   ```
   audience_mode: established | early-stage | greenfield
   detected_at: <date>
   ```

   Передать mode в подскиллы (`/segments-discover`, `/segments-unpack`, `/segments-awareness`) — они адаптируют поведение.
```

## 🔄 Resume-pattern — разбиение на сессии (см. `knowledge/state-management.md`)

**Закон:** `/audience-stage` НЕ выполняется одной сессией. Один контекст не вытянет 4-6 часов диалога + 7000 строк инструкций — модель деградирует.

**Структура сессий:** 1A (Discover ТОП-3, ~2ч) → 1B-N (Unpack каждого сегмента, ~2ч × N) → 1C-N (Awareness каждого, ~1.5ч × N). N = 1..3 в зависимости от того сколько сегментов клиент успел проработать.

**State-файлы:**
- `projects/<main>-audience/_state/pipeline-progress.md` — где в pipeline
- `office/agents/alex-marketer/agent-state.md` — что я делаю сейчас (глобальный)

**Resume Check в Pre-flight Шаг 0:** если pipeline-progress.md существует → читаем + предлагаем 3 варианта (продолжить / начать заново / завершить досрочно). Без явного выбора не идти.

**После каждого Шага:** обновить pipeline-progress.md + agent-state.md. Цитаты сохранять в файлы, не держать в контексте.

**После 3 stop+wait или 90 минут диалога:** safeguard *«Контекст забивается — `/clear` + `/audience-resume`»*.

**Полный протокол** (формат state-файлов, алгоритм, anti-patterns, связь с `/audience-status` / `/audience-check` / `/audience-resume`) — `knowledge/state-management.md`.

## Алгоритм

### Шаг 1 — Создать папку Stage 1

```bash
mkdir -p projects/<main>-audience
cp -r projects/_template-audience/. projects/<main>-audience/
#                                  ↑↑↑ точка КРИТИЧНА — копирует .gitkeep
```

⚠️ `cp -r template/* dst/` НЕ копирует скрытые файлы (`.gitkeep`). Используй `template/.` (точка в конце).

Заменить плейсхолдеры через `sed` (на macOS — `sed -i ''`, на Linux — `sed -i`):
```bash
TODAY=$(date +%Y-%m-%d)
PROJECT_NAME="<основное имя проекта>"  # например "MyProject"
MAIN_PROJECT="<main>"                   # slug, например "myproject"

find projects/<main>-audience -type f -name "*.md" -exec sed -i '' \
  -e "s/{{PROJECT_NAME}}/$PROJECT_NAME/g" \
  -e "s/{{MAIN_PROJECT}}/$MAIN_PROJECT/g" \
  -e "s/{{TODAY}}/$TODAY/g" {} \;
```

Записать в `CHANGELOG.md` создание Stage 1 (через Read+Write — см. Технические заметки внизу).

### Шаг 2 — Перенос starter-данных от Стратега

#### 2a. Если найден `diagnostic-card.md`:

Прочитать DIAG_PATH (любой из двух путей). Извлечь:

- **Описание ЦА** (как видит Стратег) — обычно в секции «Профиль клиента» или «Рыночная позиция»
- **Тип слабой стороны** (если упомянут — Воронка / Контент / Оффер / etc.)
- **Боли** которые Стратег зафиксировал
- **Гипотезы Стратега** — обычно есть отдельная секция «Гипотезы» с пометкой `[ГИПОТЕЗА]`
- **Противоречия** в данных (часто полезно для тестирования)

Записать в `projects/<main>-audience/hypotheses.md → 🧊 Backlog` минимум 3-7 starter-гипотез:

```markdown
### H1 [ICP] — «🌱 starter: основная ЦА от Стратега»
- **Источник:** {DIAG_PATH} секция X
- **Уверенность:** 30% (не валидировано данными, гипотеза эксперта)
- **Описание:** [цитата или резюме описания ЦА]
- **Тест:** валидируется в /segments-discover
- **Дедлайн:** YYYY-MM-DD (~2-3 недели)
```

H1 — становится **Key Hypothesis** (если основная ЦА от Стратега чёткая). Остальные H2-H6 — в Backlog.

#### 2b. Voc.md — отделить голос владельца от голоса ЦА:

⚠️ **Важно:** diagnostic-card часто содержит **слова САМОГО владельца про его бизнес**, а не дословные цитаты ЕГО клиентов. Это ДВА разных голоса.

Логика:
- **Цитаты ЦА** (если есть в diagnostic-card — обычно в секции «Customer voices» или с пометкой «клиент сказал») → `audience/voc.md` основная секция
- **Слова владельца про свой бизнес** (из prework, его собственных формулировок) → отдельная секция `## Голос самого {{client_name}} (из prework — для контекста)` с пометкой `#own-pain`. Они **не голос ЦА** — это материал для будущего `brand/expert-bank.md` (когда Stage 2/3 активирована).

Если в diagnostic-card НЕТ дословных цитат ЦА (что часто бывает — Стратег пишет интерпретации) — `voc.md` помечается:
```markdown
## Состояние на YYYY-MM-DD (миграция от Стратега)

⚠️ **Дословных цитат ЦА от Стратега НЕ получено.** Diagnostic-card.md содержит интерпретации, не цитаты покупателей.

**Что НУЖНО собрать в /segments-unpack:**
- Цитаты из существующих покупателей (БПСВ + 7 блоков интервью)
- Если есть переписки с лидами / диагностические звонки — транскрипты
- Комменты из соцсетей / чатов — про боли, желания, возражения
```

#### 2c. Учёт существующих audience/*.md:

Если есть `projects/<main>/audience/*.md` от Стратега (например `pains.md`, `segments.md`, `raw-segments.md`, `language.md`):

- **НЕ перезаписываем** — это работа Стратега
- **НЕ копируем содержимое** в `<main>-audience/` — будет дубль
- **Упоминаем в CHANGELOG** как «source для /segments-discover Этап 1»:
  ```markdown
  ### Что НЕ перенесено (но существует в основном проекте)
  - `projects/<main>/audience/{pains,segments,raw-segments,language}.md` — наработки Стратега. 
    Алекс прочитает их при /segments-discover Этап 1 как референс.
  ```
- **При запуске /segments-discover** — читаем эти файлы как input (это происходит уже внутри /segments-discover Этапа 1, но мы напоминаем об этом)

#### 2d. Fallback — если ничего нет от Стратега:

Короткий intake (не подменяя Стратега, только для контекста — 3 вопроса максимум):
- «Кто твой клиент сейчас (даже если общими словами)?»
- «Откуда они приходят?»
- «Что покупают / готовы купить?»

Запись в Backlog как `🌱 starter` гипотеза.

### Шаг 3 — Объяснить план

Сообщение клиенту:

```
Создал папку projects/<main>-audience/ — здесь будем распаковывать твою ЦА.
Существующий projects/<main>/ не трогаю, эта папка живёт рядом.

Перенёс из diagnostic-card N starter-гипотез в Backlog. Главная (Key) — [описание H1].

План:

1. Конкурентный анализ — 5-15 мин (опционально, но рекомендую)
   /competitors-research → разведка RU/CIS + US/EU

2. Discovery — 30-40 мин
   /segments-discover → 8 этапов: контекст → разведка → 12 гипотез
   → 6-осевой scoring → ТОП-3 сегмента в NORTH-STAR.md

3. Углубление каждого сегмента (всего 60-90 мин на 3 сегмента)
   /segments-unpack → БПСВ + 7 блоков интервью
   /segments-awareness → awareness уровни + Ctrl+X + lingvo
   → dossier.md + base.md на каждый

Между этапами — стоп, спрашиваю «продолжаем?». Без согласия не иду.

Начнём с конкурентов или сразу к Discovery?
```

**Stop+wait** — без явного согласия не запускаем.

### Шаг 4 — Двойное интервью + параллельная разведка (v1.2 переработка)

⚠️ **Новый порядок** (от старого «один большой опросник + общая разведка» отказались — давало слабые результаты): **двойное интервью + 2 параллельных субагента**.

#### 4a. Первая партия — 3 базовых вопроса (рамка) + 2 вопроса про ресурсы (выкапывание)

Алекс задаёт **5 вопросов** одной партией — но выстроены так чтобы дать рамку И начать выкапывать ресурсы клиента (его уникальность):

**Рамка (общая картина):**
1. **«Что у тебя за продукт — в одной фразе как клиенту?»**
2. **«Кто покупает в общих чертах? (профессия / ниша / стадия)»**
3. **«Что у этих покупателей реально болит? Не "проблемы вообще", а ОДНА главная боль которая их толкает к тебе.»**

**Выкапывание ресурсов (для лазера):**
4. **«В чём ты реально уникален — не на словах, а в действиях? Что ты делаешь так как никто другой? Какие у тебя есть кейсы / связи / бэкграунд которых мало у кого?»**
5. **«Кому ты УЖЕ дал больше всего пользы — расскажи 1 конкретный кейс с результатом (цифры если можешь)? И кого ты ЗАЖИГАЕШЬ сильнее всего — с кем тебе самому интересно работать?»**

Без вопросов 4-5 нельзя строить пересечение 3 кругов жирной ЦА — Алекс не будет знать что у клиента в багаже.

⚠️ **Триггер срыва scope (см. core.md «🔒 Stage Lock»).** Вопросы 1, 4, 5 могут вытащить **продуктовый материал**: клиент начнёт рассказывать про tier'ы, цены, бизнес-модель, варианты пакетов, timelines. Это **сырьё для Stage 2**, не повод туда уходить.

**Если в ответе клиента всплыли формулировки tier'ов / цен / бизнес-моделей / timelines / детальной структуры продукта:**

1. **НЕ углубляйся** — не задавай follow-up по продукту, не предлагай tier-структуру, не считай чек.
2. **Зафиксируй в `inbox/_product-hints.md`** — append дословно с тегом `[PRODUCT/OFFER/PRICE]`.
3. Скажи клиенту: *«Записал в инбокс продукта — на Stage 2 разберём. Сейчас закрываем рамку аудитории.»*
4. Возвращайся к следующему вопросу из 5 базовых.

**Stop+wait** — ждёшь ответы.

#### 4b. Вторая партия — 2-3 уточняющих (лазер по болевым точкам)

После ответов у Алекса есть **рамка + ресурсы**. Теперь он задаёт **уточняющие** чтобы найти **остроту боли + платёжность**. Они **вырастают** из ответов клиента и нацелены на пересечение 3 кругов.

Примеры по типам ответов:

**Если клиент сказал «выгорают»** → уточнение: *«выгорают от объёма команды (значит могут платить — есть команда) или потому что всё САМ (платят меньше, нет бюджета на разгрузку)?»*

**Если сказал «хотят клиентов»** → уточнение: *«идут к тебе с нуля (рынок начинающих, низкий чек) или уже что-то продают и упёрлись (рынок growth, готовы платить высокий чек)?»*

**Если сказал кейс типа "помог компании +30% выручки"** → уточнение: *«какой у них был ДО доход — 100К/мес или 1М/мес? И кто они были — фрилансер, агентство, корпорат?»*

**Если зажигают «коучи и эксперты»** → уточнение: *«какой подсегмент — психологи, продактовые коучи, бизнес-коучи? У кого из них чек выше и рынок жирнее?»*

Цель уточнений: **сузить от "примерно" до "конкретно с цифрами и платёжкой"**. **2-3 вопроса максимум.**

**Stop+wait** — ждёшь ответы.

#### 4c. Параллельный запуск 2 субагентов

После ответов на уточняющие — у Алекса есть **карта**: продукт + сегмент + боль (по версии клиента). Запускает **одновременно**:

```
┌─────────────────────────────┬─────────────────────────────┐
│  АГЕНТ A — /segment-money-map│  АГЕНТ B — /segment-hypotheses│
│  (рынок по сегментам)       │  (из ответов клиента)        │

```

Запускаются **параллельно через `Agent` tool с `run_in_background: true`**. Алекс может пока обработать ответы клиента / подготовить почву.

#### 4d. Соединение результатов → итоговая таблица

Когда оба субагента вернулись — Алекс сводит в **финальную таблицу**:

```markdown
## Money + Hypotheses — итоговая карта сегментов

| # | Сегмент-гипотеза (B) | Размер рынка (A) | Чек (A) | LTV (A) | Конкуренция (A) | Подтверждение в данных (B) | Money-score | Confidence |
|---|---------------------|-----------------|---------|---------|-----------------|---------------------------|-------------|------------|
| 1 | сегмент-A           | {N} чел         | {X} ₽   | {Y}     | MED             | N1 покупателей из M      | 8/10        | 70%        |
| 2 | сегмент-B           | {N} чел         | {X} ₽   | {Y}     | LOW             | N2 покупателей из M      | 9/10        | 60%        |
...
```

**Stop+wait** — клиент видит ВСЕ гипотезы С ДЕНЬГАМИ → выбирает ТОП-3 для распаковки.

Не «красивее всех» → а **«где быстрее всего деньги + есть подтверждение в реальных покупателях»**.

#### 4d.1 — Алекс даёт рекомендацию (НЕ прячется за таблицей)

⚠️ **Не отдавай таблицу клиенту в формате "выбирай сам".** Это нарушает принцип «вынеси выбор клиенту С РЕКОМЕНДАЦИЕЙ» (см. core.md guardrails).

После таблицы Алекс пишет 3-4 строки рекомендации:

```
Слушай, как мне видится — самый жирный сегмент сейчас это [сегмент-X].
Три причины:
- {1-я причина — пересечение какого-то из 3 кругов}
- {2-я причина — money-score / confidence}
- {3-я причина — соответствие твоим ресурсам}

Согласен взять его как hot? Или у тебя свой взгляд — поправь меня.
```

Без рекомендации Алекс просто прокладка между WebSearch и клиентом. С рекомендацией он партнёр.

**Anti-pattern:** «Вот таблица, выбирай ТОП-3» без явной позиции Алекса. Это сканер, не лазер.

**Pattern:** «Я бы взял [X], [Y], [Z] потому что [...]. Что не так?»

После выбора — переходим к Шагу 5 (`/segments-discover` со уже подготовленной рамкой) или сразу к Шагу 6 (`/segments-unpack` если ТОП-3 уже ясны).

## Stop+wait протокол

Обязательные точки остановки:
0. **Pre-flight #3** — если папка `<main>-audience/` уже есть → выбор a/b/c
1. После создания папки + переноса starter — план плюс согласие
2. После competitors-research — переходим к Discovery?
3. После Discovery (ТОП-3) — согласен с картиной?
4. После каждого `/segments-unpack {slug}` — дальше awareness?
5. После каждого `/segments-awareness {slug}` — переходим к следующему сегменту?
6. После всех 3 сегментов — финал

Внутри `/segments-discover`, `/segments-unpack`, `/segments-awareness` свои stop+wait — те не дублируем.

## ⚙️ Технические заметки для агента

### Tool-flow Read-перед-Write

Когда обновляешь существующий файл (`hypotheses.md`, `voc.md`, `CHANGELOG.md` после `cp -r`) — Write **упадёт** если файл не был сначала прочитан в этой сессии.

**Правильный порядок:**
1. `cp -r _template-audience/. <main>-audience/` (создаёт файлы)
2. `sed -i ''` для плейсхолдеров (НЕ требует Read — sed работает напрямую)
3. **Перед Write** в любой существующий файл — сначала **Read** этот файл (хотя бы офсетом)
4. Затем Write с обновлённым содержимым

**Альтернатива через Edit:** для точечных правок (одна-две строки) использовать Edit вместо Write. Edit тоже требует Read первым, но удобнее для маленьких изменений.

### Sed на macOS vs Linux

- **macOS:** `sed -i '' "s/.../.../"` (пустая строка обязательна)
- **Linux:** `sed -i "s/.../.../"` (без кавычек)

Если не уверен — проверь через `uname` и выбери правильный синтаксис.

### Кодировка и кириллица

Все шаблоны в UTF-8 с кириллицей. `sed` работает с ними нормально на macOS/Linux. Если возникают проблемы — проверь `LANG=ru_RU.UTF-8`.

## Anti-patterns

- ❌ **Перезаписать diagnostic-card.md** Стратега — нет, читаем как input, не трогаем
- ❌ **Перезаписать существующие audience/*.md** Стратега — НЕТ, они референс, не дубль
- ❌ **Создать папку без `<main>-audience` суффикса** — путать с основным проектом
- ❌ **Запустить все 3 unpack параллельно** — сегменты разные, нужна последовательность
- ❌ **Пропустить stop+wait** — клиент устаёт от 2 часов без перерыва
- ❌ **Запустить Stage 2 (продукт) автоматически** — Stage 2 это отдельный пак-надстройка
- ❌ **Смешать голос владельца с голосом ЦА в voc.md** — это два разных файла (или две явных секции)
- ❌ **Использовать `cp -r template/*`** — потеряешь .gitkeep. Используй `template/.`
- ❌ **Write в существующий файл без Read первым** — упадёт. Сначала Read
- ❌ **Произносить вслух жаргон** (`hypotheses.md`, `[ICP]`, `confidence 70%`, `Backlog`, `H1 starter`) — клиент не понимает, чувствует «бот с шаблонами». Работай с файлами **молча**, в чате — живой партнёрский язык. См. `core.md` секция «Язык клиенту — БЕЗ жаргона».
- ❌ **Строить ICP только из тёплых покупателей** — это **ошибка выжившего**. Они пришли по сарафану/связям, не из массы. Используй их как **сигнал паттерна** для горизонтального ICP, но **размер рынка / средний чек / конкуренция** считай по большой нише через `/segment-money-map`, не по 8 знакомым.
- ❌ **Структурированные таблицы и frontmatter в первом разговоре** — это для файлов (молча). В чате — живая речь, простыми предложениями.
