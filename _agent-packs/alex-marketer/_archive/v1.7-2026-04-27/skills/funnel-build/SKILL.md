# /funnel-build — Per-segment диалог воронки

> Это **внутренний движок** фазы 3 — оркестратор `/unpack-funnel` вызывает `/funnel-build` для каждого ТОП-сегмента отдельно. Один сегмент = один заход = 4 шага диалога = ~60 минут.
>
> Главное: воронка строится **из dossier и реальных данных**, не из головы. Если на каком-то шаге аргумент сводится к «я думаю что так лучше» — флаг, превращаем в гипотезу с тестом, не пишем как факт.

---

## 1. Зачем

Распаковать воронку **для одного сегмента** через 4 шага диалога:

1. **Channel-First** (Хормози) — где сегмент сейчас и какой Core Four primary
2. **Route** — маршрут трафик → LM → TW → Core по `ladder.md`
3. **Touchpoints** — последовательность касаний с темами, форматами, CTA
4. **Conversions** — метрики на каждом шаге, bottleneck, гипотезы что улучшить

На выходе — diff-предложения в `hypotheses.md → Log` по `funnel/channels.md` и `funnel/welcome.md` (плюс `scripts.md` если модуль встреч активирован).

---

## 2. Когда запускать

- **Из `/unpack-funnel`** — оркестратор вызывает в Шаге 2 для активного сегмента
- **Явно** — клиент говорит «построй воронку для сегмента {slug}», `/funnel-build {slug}`

**Не запускать самостоятельно** если ТОП-3 ещё не выбраны или оффер не валидирован — это означает что pre-flight оркестратора не отработал. Возвращаемся в `/unpack-funnel`.

---

## 3. Pre-flight checks

Скилл сам читает перед стартом — без вопросов клиенту:

| Файл | Что нужно | Если нет |
|---|---|---|
| `audience/segments/{slug}/dossier.md` | Секции 0 (профиль), 1 (БПСВ), 2 (интервью), 3 (awareness/Ctrl+X) заполнены | → стоп, `/segments-unpack` + `/segments-awareness` |
| `audience/segments/{slug}/base.md` | Желаемый контент (блок 2.7 — желаемый контент), редполитика | мягкое — продолжаем, но Шаг 3 ослаблен |
| `product/core-offer.md` | Value Equation + Grand Slam стек | → стоп, `/unpack-product` |
| `product/ladder.md` | LM → TW → Core → PM прорисованы | → стоп, `/unpack-product` |
| `brand/competitors.md` | Конкурентные механики воронок | мягкое — отмечаем gap |
| `metrics.md` | Текущие цифры по каналам | мягкое — работаем без цифр, помечаем `[нет данных]` |
| `audience/voc.md` | Реальные возражения / отказы | мягкое — без него Шаг 3 идёт на синте |
| Флаг модуля встреч | `office/modules/sales-meetings.enabled` (или аналог) | если выключен → Шаг 3 не предлагает скрипты |

---

## 4. Четыре шага диалога

### Шаг 1 — Channel-First (Хормози) — 10-15 мин

**Цель:** определить **один primary канал** под этот сегмент. Не «все 4 параллельно», не «нам нужен Instagram потому что у всех Instagram». Канал выбирается из **dossier секция awareness/Ctrl+X** — где этот сегмент сейчас физически тусуется.

**Действия:**

1. Скилл открывает `dossier.md` секцию 3 (awareness) и секцию по Ctrl+X — где ЛПР сейчас потребляет контент / ищет решения / висит в комьюнити.
2. Сводит это с Hormozi Core Four:

   | Канал | Что | Когда primary |
   |---|---|---|
   | **Warm Outreach** (1-к-1, тёплые) | Сарафан, рекомендации, личные сообщения старым контактам | Когда у клиента есть сеть с релевантными людьми |
   | **Content** (1-ко-многим, тёплые/холодные) | Посты, рассылка, YouTube, подкаст | Когда сегмент тусуется на платформах + у клиента есть голос |
   | **Cold Outreach** (1-к-1, холодные) | Холодные сообщения, email-кампании | Когда сегмент чётко идентифицируется по признакам и есть базы |
   | **Paid Ads** (1-ко-многим, холодные) | Реклама в соцсетях, поиск, ретаргет | Когда есть юнит-экономика и понятен бид |

3. Скилл предлагает **гипотезу о канале** с обоснованием **из dossier**, не из головы:

> *«По dossier этот сегмент {slug} тусуется в {где: канал/платформа/комьюнити}. Из Core Four primary ближе всего — {канал} потому что {обоснование цитатой из dossier секции 3}. Текущий CAC по этому каналу {из metrics.md или "нет данных"}. Предлагаю это как primary гипотезу. Согласен или у тебя другая интуиция?»*

4. Скилл проверяет `metrics.md` — есть ли уже цифры по этому каналу. Если есть — фиксирует факт. Если нет — помечает гипотезой с дедлайном и метрикой проверки.

5. Записывает гипотезу в `hypotheses.md` Backlog с тегом `[CHANNEL]`:

```markdown
H{N} [CHANNEL] {slug}: primary канал — {канал}
Обоснование: dossier секция 3 — «{цитата клиента или паттерн}»
Тест: запуск 2 недели, метрика — CAC ≤ {X}, лидов ≥ {Y}
Дедлайн: YYYY-MM-DD
```

**Anti-pattern (Алекс отлавливает):**

- ❌ «У нас Instagram потому что у всех Instagram» — без обоснования из dossier → **переделать**
- ❌ «Запустим всё параллельно — контент + реклама + outreach» — Хормози: 80% малых бизнесов делают одно из четырёх **до победы**, потом расширяются → **выбрать одно**
- ❌ Канал не из Core Four («комбо из всего понемногу») → **сводим к одному из четырёх**

#### Stop+wait после Шага 1

Скилл показывает клиенту:
- Где сегмент сейчас (цитата из dossier)
- Primary канал + обоснование
- Текущий CAC если есть, или гипотеза с тестом
- Что **НЕ** primary в этой итерации (явно — чтобы не размазывать)

Ждёт «ок / другой канал / докрути обоснование». Без апрува — Шаг 2 не запускается.

---

### Шаг 2 — Route (маршрут) — 15-20 мин

**Цель:** прорисовать **маршрут от трафика до Core** по этому сегменту, согласно `ladder.md`. Что человек видит первым → как переходит к LM → как платит первый рубль (TW) → как доходит до Core.

**Действия:**

1. Скилл читает `ladder.md` — берёт лестницу LM → TW → Core → PM.

2. Для **этого сегмента** проверяет:
   - Какой LM из лестницы релевантен под его боли / awareness / темы из dossier?
   - Какой TW (низкоценовой вход, обычно 500-3000р) ведёт к Core?
   - Какие **bridges** между ступенями — что переводит человека с LM на TW и с TW на Core?

3. Если LM/TW из лестницы **не подходят сегменту** — Алекс честно говорит:

> *«Текущий LM "{название}" из ladder.md был придуман под средне-обобщённого клиента. Для {slug} он слабоват: dossier говорит этот сегмент сейчас на L{уровень awareness} — он не понимает зачем ему "PDF про X". Ему нужно "{формат под awareness — например, диагностика / 5-минутный калькулятор / разбор кейса}". Это гипотеза `[LM]` — заводим тест?»*

4. Скилл прорисовывает маршрут:

```
{Primary канал из Шага 1}
   ↓ (хук, креатив, тема контента)
{LM} — {название, формат, обещание, для чего сегменту}
   ↓ (bridge — благодарственная страница / welcome-email / DM)
{TW} — {название, цена, что включено}
   ↓ (bridge — апсейл письмо / упоминание в TW-материале / личный контакт)
{Core} — {core-offer.md}
   ↓ (опционально PM)
{PM}
```

5. **Bridges критичны.** Алекс не пропускает их — это места где клиенты «теряются между ступенями». Каждый bridge явно описывается:

| Переход | Bridge | Тип | Что в нём |
|---|---|---|---|
| LM → TW | Welcome-email серия | Текст, 3 письма за 7 дней | Раскрытие проблемы → кейс → оффер TW |
| TW → Core | Финальный модуль TW | Внутри TW-материала | Естественный мост: «если хочешь полное решение → Core» |

**Anti-pattern:**

- ❌ Лестница «как у всех» — копия конкурентов без проверки против dossier этого сегмента → **переделать**
- ❌ Прыжок через ступень («сразу с LM на Core») — теряем тех кто не готов платить полную цену → **либо честно убрать TW, либо вернуть его**
- ❌ Bridges «само собой подразумевается» — нет — где конкретно сообщение, формат, текст? → **прорисовать**

#### Stop+wait после Шага 2

Скилл показывает клиенту:
- Маршрут схемой (канал → LM → TW → Core)
- Каждый шаг с обоснованием от dossier
- Где из ladder.md что-то требует адаптации под сегмент → гипотеза с дедлайном
- Bridges явно описаны

Ждёт «ок / поменяй LM / поменяй TW / убери TW». Без апрува — Шаг 3 не запускается.

---

### Шаг 3 — Touchpoints (касания) — 15-25 мин

**Цель:** прорисовать **последовательность сообщений** в воронке. Не «отправим welcome-цепочку», а **сколько касаний, темы, форматы, CTA, период**.

**Норма по индустрии (из `knowledge/classics-compact.md`):** 7-12 касаний от первого до Core. Меньше 5 — не успевает построиться доверие, больше 12 — выгораем интерес.

**Действия:**

1. Скилл смотрит на awareness уровень сегмента (dossier секция 3):
   - L1-L2 (problem aware) — нужно **больше касаний** (7-12), сначала образование/проблематизация, потом продукт
   - L3 (solution aware) — **средняя цепочка** (5-7 касаний), сравнения и доказательства
   - L4-L5 (product aware / most aware) — **короткая цепочка** (3-5 касаний), оффер + гарантия + CTA

2. Скилл строит таблицу касаний:

| # | Когда | Канал | Формат | Тема / Хук | CTA |
|---|---|---|---|---|---|
| 1 | День 0 (момент подписки) | Email + DM в TG | Welcome-письмо | «Спасибо за {LM}, вот как это работает» | Прочитать материал |
| 2 | День 1 | Email | Кейс-история | «{Имя клиента из voc} — как сделал {результат}» | Опционально — заявка |
| 3 | День 3 | Email + push | Разбор боли | «3 причины почему {боль из dossier} не уходит» | Прочитать |
| 4 | День 5 | Email | Прямой оффер TW | «Если хочешь начать — вот {TW}» | Купить TW |
| ... | ... | ... | ... | ... | ... |

3. **Каждое касание привязано к dossier:**
   - Тема — из боли/желания сегмента (цитата из dossier секции 1.2 или 2.7)
   - Хук — из живого языка сегмента (lingvo-мап, секция 4)
   - CTA — соответствует уровню awareness

4. Если активирован **модуль встреч** (`/marketer-enable-meetings` запущен):
   - Алекс добавляет в цепочку **созвон-точку** (обычно после касания 3-4 для тёплого сегмента)
   - Упоминает что **скрипты диагностики и банк возражений** для созвона лежат в `funnel/scripts.md` и заполняются отдельно (или генерятся по карточкам клиентов из `customers/`)
   - Возражения для скрипта берутся из **реальных карточек клиентов** + `audience/voc.md`, не из головы

5. Если **модуль встреч НЕ активирован**:
   - В цепочке только **текстовые касания** (письма / посты / DM / push)
   - Возражения, которые в живой беседе закрылись бы продажником, **закрываются в текстах**
   - Алекс открывает `audience/voc.md`, ищет тег `#objection`, и встраивает **закрытие возражения** в соответствующее касание (например, «дорого» → касание 5 «как окупается за 30 дней с цифрами»)
   - Если в `voc.md` пусто — Алекс честно говорит: «возражения не накоплены, цепочка пишется на гипотезах, надо собрать voc минимум 10 цитат за 2 недели для калибровки»

**Anti-pattern:**

- ❌ «Продаём в первом письме» — выгораем интерес, особенно для L1-L2 awareness → **увеличить дистанцию до оффера**
- ❌ 3+ касания подряд про **один и тот же продукт** → **разбавить** контентом, кейсами, разборами
- ❌ Касания без CTA («просто пост») — это тратит touch без движения → **добавить CTA каждому касанию, даже мягкий**
- ❌ Темы из головы («у меня в этом письме будет про мотивацию») без привязки к боли/желанию из dossier → **переделать с цитатой из dossier**
- ❌ Скрипты созвона предлагаются при выключенном модуле встреч → **убрать, оставить только тексты**

#### Stop+wait после Шага 3

Скилл показывает клиенту:
- Таблицу касаний (7-12 строк)
- Привязку каждого касания к dossier (боль / тема / язык)
- При активном модуле встреч — куда вставлен созвон + ссылка на `scripts.md`
- При выключенном — какие возражения закрываются в каких касаниях из `voc.md`

Ждёт «ок / убери касание N / добавь касание про X / переделай тему касания M». Без апрува — Шаг 4 не запускается.

---

### Шаг 4 — Conversions (метрики) — 10-15 мин

**Цель:** наложить **цифры** на построенный маршрут. Где сейчас, где целимся, где главная просадка.

**Действия:**

1. Скилл строит таблицу метрик по этому сегменту:

| Этап | Текущая | Целевая | Бенчмарк по нише | Просадка |
|---|---|---|---|---|
| Трафик → подписка на LM | {из metrics.md или [нет данных]} | {гипотеза} | {из knowledge/} | {если разрыв > 30%} |
| LM → оплата TW | ... | ... | ... | ... |
| TW → апсел Core | ... | ... | ... | ... |
| Касания → заявка / оплата Core | ... | ... | ... | ... |

2. Если `metrics.md` пустой / частично пустой — **Алекс не выдумывает цифры**. Помечает `[нет данных]` и формулирует гипотезу:

```markdown
H{N} [METRIC] {slug}: текущая конверсия LM→TW неизвестна.
Тест: запустить трекинг (UTM + thank-you-page event) на 2 недели,
снять базовую цифру для сравнения с бенчмарком 8-12%.
Дедлайн: YYYY-MM-DD.
```

3. **Bottleneck-анализ (Голдрат) — главное в этом шаге.** Алекс ищет **одно** место где течёт сильнее всего:

> *«Смотри. По данным которые есть — у тебя на этапе LM→TW конверсия 2%. Норма по нише 8-12%. Это **главный bottleneck** этого сегмента. Остальные этапы средние — там не теряем больше чем нормально. Вот гипотеза что улучшить — H{N}.»*

4. Гипотеза в `hypotheses.md` с тегом `[FUNNEL]`:

```markdown
H{N} [FUNNEL] {slug}: bottleneck LM→TW (2% при норме 8-12%)
Рабочая причина: TW позиционируется как мини-курс, ЦА на L2 awareness ждёт
формат «попробовать» — диагностика или PDF с быстрым выигрышем.
Тест: переписать описание TW под формат «5 вопросов и я говорю что у тебя
сломано», запуск 2 недели.
Метрика успеха: конв ≥ 6%. Метрика провала: < 5% — откат, ищем другой формат.
Дедлайн: YYYY-MM-DD.
```

5. **Правило:** одна-две гипотезы по bottleneck — **не пять**. Голдрат: чинить узкое место, не везде. Если предложений больше — приоритезировать, остальное в Backlog.

**Anti-pattern:**

- ❌ «Везде есть просадка, надо чинить всё» → **выбрать одно главное узкое место**
- ❌ Bottleneck без цифр («чувствую что течёт здесь») → **либо найти цифру, либо помечать `[нет данных, гипотеза N%]` и тестировать**
- ❌ Метрика успеха без даты / без цифры провала → **уточнить — без них это надежда, не тест**

#### Stop+wait после Шага 4

Скилл показывает клиенту:
- Таблицу метрик (current → target → бенчмарк → просадка)
- Главный bottleneck с обоснованием цифрой
- 1-2 гипотезы что улучшить (с тестом, метрикой, дедлайном)
- Backlog предложений (что не вошло в активные)

Ждёт «принимаю / переделай гипотезу / найди другой bottleneck». После апрува — `/funnel-build` закрывается, управление возвращается в `/unpack-funnel` Шаг 3 (stop+wait после сегмента + переход к следующему).

## 5. Output contract

| Куда | Что | Когда |
|---|---|---|
| `hypotheses.md → Backlog` | Гипотезы с тегами `[CHANNEL]`, `[LM]`, `[FUNNEL]`, `[METRIC]` | По мере появления на каждом шаге |
| `hypotheses.md → Active` | Перевод из Backlog после согласия клиента (макс 5 одновременно) | После Шага 4 |
| `hypotheses.md → Log` | diff-предложения по `funnel/channels.md` (этот сегмент → канал) и `funnel/welcome.md` (касания) | Финал `/funnel-build` |
| `funnel/channels.md` | Канальный контракт сегмента | **Только после `/accept H{N}` владельцем** |
| `funnel/welcome.md` | Касания этого сегмента | **Только после `/accept H{N}` владельцем** |
| `funnel/scripts.md` | Скрипты созвона + банк возражений | **Только при активном модуле встреч + `/accept H{N}`** |
| `audience/voc.md` | Возражения / отказы пойманные по ходу | append-only **сразу** |
| `metrics.md` | AUTO-секции если появились новые цифры | append AUTO-зоны |

**Закон голоса:** скилл **не перезаписывает** `funnel/*.md` напрямую. Только diff в Log → `/accept` от владельца.

## 6. Anti-patterns (сводный список)

- **Унификация воронки для всех сегментов** — каждый сегмент должен иметь свой маршрут. Если три сегмента ведут одинаково через один LM и один TW — это сигнал что мы либо плохо распаковали сегменты, либо ленимся per-segment
- **Игнорирование bottleneck** — рисуем красивую воронку, не смотрим где течёт. Без Голдрата воронка = декорация
- **Скрипты «из головы»** — без реальных возражений из `voc.md` и карточек клиентов скрипт = фантазия. Возражения снизу
- **3+ касания подряд про один и тот же продукт** — выжигаем интерес, особенно у L1-L2 awareness. Разбавляем контентом / кейсами / разборами
- **Канал «как у всех»** без обоснования из dossier — копия конкурентов без проверки релевантности сегменту
- **Прыжок через ступень лестницы** — сразу с LM на Core. Теряем тех кто не готов платить полную цену
- **Bridges «само собой»** — между ступенями ничего нет, переход не описан. Bridges — это сообщения, а не магия
- **Метрики без цифры провала** — «целимся в 6%» без «если меньше 5% — откат» — это надежда, не тест
- **Скрипты созвона при выключенном модуле встреч** — `funnel/scripts.md` не нужен, возражения копятся в `voc.md` и закрываются текстами

## 9. Принципы скилла (короткий чек-лист)

- **Один сегмент за заход** — `/funnel-build` всегда per-segment, не пытаемся обобщать
- **Channel-First из dossier** — канал primary обосновывается данными о сегменте, не «как у всех»
- **Bottleneck — один** — Голдрат: главное узкое место одно, чинить его
- **Касания привязаны к dossier** — тема, хук, CTA из болей/желаний/языка сегмента
- **Возражения снизу** — `voc.md`, карточки клиентов, реальные цитаты, не из головы
- **Скрипты только при модуле встреч** — иначе возражения закрываются текстом
- **Закон голоса** — diff в Log, `/accept` владельца, никаких прямых перезаписей
- **Stop+wait после каждого шага** — без апрува следующий шаг не запускается
- **Цифры или пометка `[гипотеза]`** — не выдумывать ни одной метрики
- **Em-dash `—`, не `--`** в любых текстах файлов
