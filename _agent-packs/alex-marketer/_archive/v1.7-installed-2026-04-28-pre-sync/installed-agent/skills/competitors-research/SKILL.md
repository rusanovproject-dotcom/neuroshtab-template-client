# /competitors-research — Скилл №0 пайплайна распаковки ЦА

## Цель

Собрать конкурентную разведку, без которой Скилл №1 (`/segments-discover`)
на Этапе 8 не может проставить ось «Whitespace vs Red Ocean» и обоснованно
отделить голубой океан от перегретого рынка. Без этого скилла любой ответ
«наш продукт уникален» — это спекуляция.

Этот скилл — **первый** в пайплайне. Запускается **синхронно** изнутри
Скилла №1 на старте Этапа 0 (см. план 4.1) — после первого вопроса клиенту
«3 конкурента + ниша + регион», ДО оставшихся 8 вопросов. Скилл №1 ждёт
результата 5-15 минут и продолжает диалог уже с готовой разведкой.

## Когда запускать

| Триггер | Источник |
|---|---|
| **Sync из Скилла №1** | Этап 0 `/segments-discover` зовёт автоматически |
| **Manual ручной запуск** | Маркетолог сам пишет `/competitors-research` |
| **Cron раз в месяц** | Напоминание 1-го числа 09:00 — `/competitors-research --refresh` |
| **Внеплановый триггер** | «Слышал у X запустили Y», новый игрок появился, конкурент сменил позиционирование |

При `--refresh` — текущие данные переносятся в `_archive/YYYY-MM/`,
оба субагента запускаются заново.

## Вход

**Минимальный** (по плану 4.2):
- Топ-3 конкурента (имена брендов + ссылки на лендинги или telegram-каналы)
- Ниша / тема (одна строка)
- Регион (RU/CIS — обычно дефолт)

**Параметры:**
- `--include-global` (default = `true`) — включает субагент B (US/EU). Можно
  отключить для микро-ниш где западные кейсы нерелевантны.
- `--refresh` — режим обновления, архивирует старые данные.
- `--slug <slug>` — slug проекта для папки `intel/competitors-{slug}/`.
  Если не передан — берётся из имени активного проекта.

**Опциональный:**
- Известные «эталоны» (бренды, на которые клиент хочет быть похож)
- Антипатии (бренды, на которые точно НЕ быть похожим)

**Live-диалог при недоборе входа** (по T-Dialog-8):
- Клиент: «у меня уникальный продукт, конкурентов нет» → задай:
  «Пользователь твоего продукта **сейчас как-то** решает проблему. Не у тебя —
  у кого? Делает руками? Платит коучу? Сидит в Excel? Это и есть конкуренты —
  даже если они не похожи на твой продукт. Дай 3 примера: «Иван сейчас делает X»,
  «Анна сейчас платит Y», «Алексей сейчас просто терпит». Это и есть твоя
  whitespace-карта.»
- Клиент: «введите название ниши» — НЕ спрашивай шаблонно. Спроси:
  «Если бы клиент гуглил решение твоей задачи — какие 3 запроса он бы вбил?»
  (живой формат вместо абстракции, T4.2 живого диалога).

## Архитектура — 2 субагента в одном Task-вызове

По плану 4.3: скилл стартует, разворачивает входные данные, запускает обоих
субагентов в **одном Task-вызове**. Они выполняются последовательно или
внутренне-параллельно (это деталь реализации, не контракт).

Скилл возвращает результат когда **оба** субагента завершились.

Если `--include-global=false` — субагент B пропускается, в `global/` кладётся
файл `_disabled.md` с пометкой «отключён параметром --include-global=false».

```
/competitors-research
        │
        └─ Task tool (одно вызов)
              ├─ Субагент A — Локальный рынок RU/CIS
              └─ Субагент B — Западный рынок US/EU (если --include-global=true)
```

### Субагент A — Локальный рынок (RU/CIS)

**Задача:** найти где деньги в нашем регионе.

**Что ищет** (по плану 4.3):

1. **Лендинги топ-3 конкурентов** — извлекает: offer (что обещают), price
   (какие тарифы), guarantees (какие гарантии), target (на кого
   позиционируются), hero-блок дословно.
2. **VoC (voice of customer)** — отзывы на сайте, комментарии в каналах,
   что хвалят, что ругают. Минимум 5 цитат каждого типа.
3. **Каналы** — где они появляются: Telegram (имя канала, число
   подписчиков), форумы (Habr, vc.ru, Pikabu), конференции (где выступают),
   профильные чаты.
4. **Цены/тарифы карта** — таблица: «конкурент | тариф | цена | что
   включено | гарантия». 3-5 позиций.
5. **Whitespace локальный** — что конкуренты НЕ делают, но клиенты просят
   (поиск по жалобам в отзывах).
6. **Сегментный срез + БПСВ-паттерны** — для каждого конкурента: на какой
   сегмент работает (3W: кто/что обещает/зачем нужно), какие 5-7 болей решает
   (буллеты дословно с лендинга), 5-7 хотелок (дословно), 3-5 возражений
   которые снимает. Плюс сводный срез по нише: топ-7 болей повторяющихся у 3+
   конкурентов, топ-7 хотелок, топ-5 возражений, карта сегментов которые
   обслуживаются. **Это сырьё для `/segments-unpack` Шаг БПСВ** — Алекс
   использует эти срезы чтобы сгенерировать 10 вариантов болей/потребностей
   /страхов/возражений на выбор клиенту, а не спрашивать «дай 2-3».

**Источники (приоритет):** WebSearch по бренду + «отзывы»; WebFetch
лендингов; WebSearch по теме + «vc.ru»; WebSearch по теме + «forum»;
WebFetch Telegram-страниц через `t.me/s/<channel>`.

**Output (физические файлы):**
- `intel/competitors-{slug}/local/landings.md` — карта лендингов по блокам
- `intel/competitors-{slug}/local/voc.md` — голос клиента, классификация
  (что хвалят / что ругают / чего не хватает), ≥15 цитат с `source:`
- `intel/competitors-{slug}/local/channels.md` — где они в эфире
- `intel/competitors-{slug}/local/prices.md` — карта цен
- `intel/competitors-{slug}/local/whitespace.md` — что не закрыто
- `intel/competitors-{slug}/local/segments-bpsv-cuts.md` — **сегментный срез
  + БПСВ-паттерны** (на кого работают, какие боли/хотелки/возражения
  выкатывают на лендинге дословно). Сырьё для `/segments-unpack` Шаг БПСВ.

### Субагент B — Западный рынок (US/EU)

**Задача:** найти что переносимо в RU.

**Что ищет** (по плану 4.3):

1. **Топ-кейсы по нише** — 3-5 западных аналогов (например, для
   AI-делегирования: Lindy, Adept, Cognition).
2. **Уникальные механики и модели** — что они продают (subscription,
   one-time, usage-based, freemium, community-led).
3. **Принципы** — что они делают по-другому против локального рынка.
   Не «они тоже делают курсы», а «они делают community-driven cohort
   с peer learning».
4. **Перенос в RU/CIS** — какие 3 механики из найденного применимы
   у нас (с обоснованием).

**Жёсткий фильтр (план 4.3):** только где живые деньги (есть фандинг или
подтверждённая выручка), реальные кейсы (имена клиентов), не маркетинговая
шелуха (Medium-статьи и Twitter-треды без backing).

**Источники:** WebSearch по теме на английском; WebFetch ProductHunt;
WebFetch Crunchbase (через open data); WebSearch + «case study» /
«pricing» / «how it works».

**Output:**
- `intel/competitors-{slug}/global/cases.md` — топ западные кейсы
- `intel/competitors-{slug}/global/mechanics.md` — уникальные механики
  (3-5 штук, формат «что делают / почему работает / как переносимо»)
- `intel/competitors-{slug}/global/principles.md` — принципы организации
  продукта

## Output структура — детально

По плану 4.4:

```
intel/competitors-{slug}/
├── INDEX.md                  ← мастер-индекс с верхнеуровневой выжимкой
├── local/
│   ├── landings.md           (топ-3 лендинга, разобрано по блокам)
│   ├── voc.md                (голос клиента, ≥15 цитат с source)
│   ├── channels.md           (где конкуренты вещают)
│   ├── prices.md             (карта тарифов)
│   └── whitespace.md         (что не закрыто на локальном рынке)
├── global/
│   ├── cases.md              (3-5 западных аналогов с раскрытием)
│   ├── mechanics.md          (3 механики применимые у нас)
│   └── principles.md         (принципы конкурентов которые стоит знать)
├── _archive/
│   └── YYYY-MM/              (предыдущая версия при --refresh)
├── manual-research-template.md  (создаётся в fallback-режиме)
└── meta.json                 (дата генерации, источники, версия скилла)
```

### INDEX.md — верхнеуровневая выжимка (≤ 60 строк)

Содержит:

- **3 механики с западного рынка** которые стоит украсть (или знать что НЕ
  украсть)
- **3 жалобы клиентов на локальных конкурентов** (whitespace для нас)
- **3 ценовых якоря** (что считается дорого / средне / дёшево в нише)
- **Балл Whitespace vs Red Ocean** — целое число 1-5 с обоснованием

Шкала Whitespace (для оси scoring Скилла №1, Этап 8):

| Балл | Что значит |
|---|---|
| 5 | Голубой океан — ниша пустая, никто не закрывает |
| 4 | Несколько слабых игроков, рынок не сформирован |
| 3 | Конкуренция есть, но whitespace явный (default при partial) |
| 2 | Сильные игроки, узкая дыра |
| 1 | Red ocean — перегретый рынок, все говорят одно |

### Frontmatter каждого выходного файла

Все `*.md` файлы в `local/` и `global/` начинаются единым frontmatter:

```yaml
---
slug: {slug}
generated: 2026-04-25T14:32:00+03:00
source: subagent-A | subagent-B | manual
version: 1
status: ok | partial | manual_pending
---
```

### meta.json

```json
{
  "slug": "{slug}",
  "generated_at": "ISO-8601",
  "skill_version": "1",

  "subagent_a": { "status": "ok|partial|failed", "sources_count": 12 },

```

## Шаги работы

По плану 4.3:

### Шаг 1 — Прочитать вход

- Slug проекта (или вычислить из имени проекта)
- Топ-3 конкурента + ссылки
- Ниша + регион
- Параметры `--include-global`, `--refresh`

Если входа недостаточно — задай **один** живой вопрос и жди ответа (НЕ
вываливай чек-лист).

### Шаг 2 — Создать структуру папок

```bash
mkdir -p intel/competitors-{slug}/local
mkdir -p intel/competitors-{slug}/global
```

При `--refresh` — сначала `mv intel/competitors-{slug}/{local,global,INDEX.md}
intel/competitors-{slug}/_archive/YYYY-MM/`.

### Шаг 3 — Запустить субагент A через Task tool

Один Task-вызов с `subagent_type: general-purpose` и промтом из секции
«Промпт для субагента A» ниже. Работает 5-10 минут.

### Шаг 4 — Запустить субагент B (если `--include-global=true`)

В **том же** Task-вызове следом за A (последовательно), либо в отдельном
Task если поддерживается параллелизм — но контракт остаётся: один
оркестрационный Task на скилл (см. T0.1).

Если `--include-global=false` — пропустить, создать
`intel/competitors-{slug}/global/_disabled.md` с пометкой
«отключён параметром --include-global=false».

### Шаг 5 — Собрать INDEX.md

Из output обоих субагентов извлечь:
- 3 механики (из `global/mechanics.md`)
- 3 жалобы (из `local/voc.md` секция «что ругают»)
- 3 ценовых якоря (из `local/prices.md` — нижний / средний / верхний)
- Список whitespace-кандидатов (из `local/whitespace.md`) с короткими
  именами для MAP.md

Уложить в ≤60 строк. Не вываливать сырьё — выжимка executive-уровня.

### Шаг 6 — Посчитать балл Whitespace 1-5

По шкале выше. Это и есть значение для 6-й оси scoring Скилла №1
(Этап 8, см. план 3.6). Балл проставляется в INDEX.md и в `meta.json`.

### Шаг 7 — Передача в MAP.md

Каждая whitespace-гипотеза из `local/whitespace.md` дублируется одной
строкой в `audience/segments/MAP.md` со ссылкой
`источник: intel/competitors-{slug}/local/whitespace.md` (требование T0.7).

### Шаг 8 — Вернуть результат

Скиллу №1 (если запущен из него) — путь `intel/competitors-{slug}/INDEX.md`
и балл Whitespace для `discover-workspace.md`.

Маркетологу (если ручной запуск) — короткий чат-ответ:
> «Готово. Балл Whitespace = N. INDEX тут: `intel/competitors-{slug}/INDEX.md`.
> Если что-то режет глаз — скажи, допишу.»

## Acceptance criteria

По плану 4.5. Скилл считается успешно отработавшим, если:

- [ ] Файлы `local/` и `global/` созданы и не пустые (>200 байт каждый)
- [ ] `INDEX.md` существует и ≤60 строк, содержит балл Whitespace 1-5
      с обоснованием
- [ ] В `local/voc.md` минимум **15 цитат** с указанием `source:` для каждой
      (T0.4) — дословно, в кавычках, не пересказ
- [ ] В `global/mechanics.md` минимум **3 механики** с раскрытием «что
      делают / почему работает / переносимо ли»
- [ ] В `local/whitespace.md` минимум **3 явных whitespace-сегмента**
      в формате «Никто не закрывает: {X}. Сигнал: {цитата/факт}.
      Гипотеза сегмента: {имя}» (T0.5)
- [ ] В `local/segments-bpsv-cuts.md` **по каждому конкуренту** заполнены:
      сегмент (3W) + 5-7 болей дословно с лендинга + 5-7 хотелок дословно
      + 3-5 возражений. Плюс **сводный срез по нише**: топ-7 болей повторяющихся
      у 3+ конкурентов / топ-7 хотелок / топ-5 возражений / карта обслуживаемых
      сегментов. Это сырьё для `/segments-unpack` — без него скилл №2
      не сможет сгенерировать 10 вариантов БПСВ корректно.
- [ ] `meta.json` содержит дату, перечень источников, версию
- [ ] Минимум 3 топ-конкурента в `local/landings.md` с заполненными
      блоками offer + price + hero
- [ ] Жёсткий фильтр пройден — каждая запись объясняет «почему это про
      деньги / механики / принцип»
- [ ] В `MAP.md` добавлена ≥1 ссылка вида `intel/competitors-{slug}/`
      (T0.7)

Если хоть один пункт не выполнен — скилл возвращает результат с пометкой
`[partial]` в INDEX.md, маркетолог видит дыру и решает что делать.

## Жёсткое резание мусора

По плану 4.7. Аналитик возвращается **только с тем, что интересно**:

- **Где деньги** — конкретные цены, тарифы, ценовые якоря.
- **Где принцип** — что отличает работающего конкурента от мертвечины.
- **Где механика** — какая фича / процесс / формат продукта реально
  переносим.

**Что НЕ берём (выбрасываем сразу):**

| Тип мусора | Признак | Действие |
|---|---|---|
| Маркетинговая шелуха без цифр | Слова «современный», «удобный», «инновационный», «революционный» в описании | Вычёркиваем (T0.2: эти слова запрещены в output) |
| Лендинги без цены | Только «оставьте заявку» | Помечаем `pricing: hidden`, цитата идёт в whitespace «не показывают цену» |
| Кейсы без указания результата | «Помогли клиенту вырасти» без цифр | Не цитируем |
| Twitter-треды от не-практиков | Theorizing без backing | Игнорируем |
| Скриншоты курсов которых нет в продаже | Платформа продажи мертва, чек-аут не работает | В `_dead.md` отдельным файлом, не в основной канон |
| Конкурент которого нашли но он не живой | Сайт не обновлялся 12+ мес, нет постов в TG, нет отзывов после 2024 | Не включаем в топ-3 |

**Аналитик НЕ возвращается с фразой** «вот лендинги конкурентов» — возвращается с
«вот 3 механики которые работают и стоят денег, вот 5 жалоб клиентов
конкурентов, вот ценовая карта от {min} до {max}».

## Fallback без WebSearch / WebFetch

По плану 4.6.bis. Если у клиента не подключены MCP с веб-поиском —
субагенты не смогут собрать данные автоматически. Скилл:

1. **Возвращает `[partial]`** в INDEX.md с баллом Whitespace = **3**
   (нейтральный) и обоснованием «WebSearch недоступен, ось требует
   ручной валидации».
2. **Создаёт шаблон** `intel/competitors-{slug}/manual-research-template.md`
   со структурой:
   - Список лендингов конкурентов на ручную проверку (имена + URL, пусто
     для заполнения)
   - Таблица цен (пустая колонками: конкурент / тариф / цена /
     что включено / гарантия)
   - Список TG-каналов (пустой)
   - Вопросы клиенту для самостоятельного research:
     - «Открой 3 лендинга конкурентов и скопируй сюда главный hero-блок»
     - «Какие 3 цены ты видишь у них чаще всего?»
     - «Что они обещают чего ты НЕ обещаешь?»
3. **Маркетолог** проводит разведку вручную, заполняет шаблон. После
   этого вручную обновляет INDEX.md актуальным баллом Whitespace.
