# /unpack-funnel — Скилл фазы 3 пайплайна распаковки

> Это **третья фаза** мета-скилла `/unpack-project` — после распакованной ЦА (фаза 1) и собранного оффера (фаза 2). Здесь Алекс строит **маршрут от первого касания до сделки** для каждого ТОП-сегмента: где сегмент тусуется, какой канал primary, какие лид-магниты и tripwire ведут к Core, какая последовательность касаний нужна, где сейчас протекают деньги.
>
> Ключевая идея: **per-segment**. Не «общая воронка для всех» — у каждого ТОП-сегмента свой маршрут. Унификация на этом шаге = первая дыра по которой течёт бюджет.

---

## 1. Зачем

Получить **per-segment воронки** для ТОП-сегментов из `NORTH-STAR.md`:
- какой Core Four канал primary под сегмент (Хормози: контент / реклама / outreach / нетворкинг),
- маршрут трафик → лид-магнит → tripwire → Core (по `ladder.md`),
- последовательность касаний с темами / форматами / CTA,
- метрики на каждом шаге и **bottleneck** — где главная просадка (Голдрат),
- предложения diff в `funnel/channels.md`, `funnel/welcome.md` (и при модуле встреч — `funnel/scripts.md`).

На выходе — все ТОП-сегменты прошиты воронкой, гипотезы по каналам и касаниям лежат в `hypotheses.md → Log`, владелец принимает командой `/accept H{N}`.

---

## 2. Когда запускать

- **Из мета-скилла `/unpack-project`** — фаза 3, после `/unpack-product`
- **Явно** — клиент говорит «распакуй воронку», «по сегменту воронка», «journey», «где у меня течёт»
- **При появлении нового ТОП-сегмента** — после `/segments-discover` распаковали новый hot, надо построить ему маршрут
- **При смене оффера** — обновился Core, маршрут к нему может перестать работать

**Не запускать:**
- Если ТОП-3 ещё не выбраны → `/segments-discover`
- Если `core-offer.md` пустой или не валидирован → `/unpack-product`
- Для мелких корректировок одного канала / письма → это `/marketer-revision` или прямой diff в `hypotheses.md`

## 4. Алгоритм

### Шаг 1 — Прочитать NORTH-STAR.md и выбрать активный сегмент

Скилл читает `audience/segments/NORTH-STAR.md`. Видит ТОП-3.

- Если ТОП-3 = 1 сегмент → автоматически идёт по нему
- Если 2-3 сегмента → спрашивает клиента живой формулировкой:

> *«У тебя в ТОП-3 сейчас три сегмента — hot {имя}, warm {имя}, cold {имя}. Воронку строим **по одному**, не общую. С какого начинаем — с hot или хочешь иначе? Я бы пошёл с hot — там быстрее всего деньги увидим.»*

**Принцип:** один сегмент за заход. Не пытайся параллельно распаковать hot+warm+cold — смешаются каналы, потеряешь фокус.

---

### Шаг 2 — Запуск `/funnel-build` для активного сегмента

Скилл вызывает `/funnel-build` с slug сегмента. Подскилл проводит **4-шаговый диалог**:

| Шаг | Что | Время |
|---|---|---|
| 1. **Channel-First** (Хормози) | Где сегмент сейчас, какой Core Four primary, текущий CAC | 10-15 мин |
| 2. **Route** | Маршрут трафик → LM → TW → Core по `ladder.md` | 15-20 мин |
| 3. **Touchpoints** | Последовательность касаний (writeups, email, посты, созвоны) | 15-25 мин |
| 4. **Conversions** | Метрики на каждом шаге, bottleneck, гипотезы что улучшить | 10-15 мин |

Внутри `/funnel-build` свои stop+wait после каждого шага. Подробности — `funnel-build/SKILL.md`.

**Output `/funnel-build`:** в `hypotheses.md → Log` — diff-предложения по `funnel/channels.md` (этот сегмент → этот канал) + `funnel/welcome.md` (последовательность касаний). Если модуль встреч активирован — diff и для `funnel/scripts.md`.

---

### Шаг 3 — Stop+wait после каждого сегмента

После закрытия `/funnel-build` для одного сегмента — **жёсткий stop+wait**:

> *«По сегменту {hot-имя} воронка распакована. Channel primary — {канал}, маршрут — LM "{название}" → TW "{название}" → Core, касаний 7. Главный bottleneck — {где течёт}. Diff лёг в `hypotheses.md` под H{N1}-H{N3}. Принимаешь? Или хочешь докрутить какой-то шаг? Дальше — следующий сегмент или закроем фазу здесь?»*

Без явного «ОК, фиксируем + следующий» / «давай только этот, остальные потом» — Шаг 4 не запускается.

**Варианты решения клиента:**

1. **«Принимаю + следующий сегмент»** → клиент делает `/accept H{N}` для каждой принятой гипотезы, скилл переходит к Шагу 4 (повторяет цикл для warm)
2. **«Принимаю только этот, остальные позже»** → скилл закрывает фазу 3 здесь, переходит к Шагу 5 (bottleneck-анализ только по этому сегменту)
3. **«Не принимаю, переделай»** → откат внутри `/funnel-build`, повторение диалога с правками

---

### Шаг 4 — Повтор цикла для следующих сегментов

Если клиент сказал «продолжаем» — Шаг 2 + Шаг 3 для warm, потом для cold.

**Правило:** скилл сам **не торопит** клиента. Если после hot-сегмента видно усталость («давай завтра», «голова уже не варит») — фиксирует прогресс в `hypotheses.md`, оставляет open thread, клиент возвращается через `/unpack-funnel` к следующему сегменту в другой сессии.

---

### Шаг 5 — Bottleneck-анализ по всей воронке (Голдрат)

После того как **все ТОП-сегменты распакованы** (или клиент закрыл фазу раньше) — скилл делает **сводный bottleneck-анализ**.

**Цель:** не «у нас 3 воронки и в каждой свои дыры», а «вот **одна** точка где деньги протекают суммарно сильнее всего по всем сегментам».

**Действия:**

1. Скилл собирает таблицу — по сегментам × этапам воронки:

| Этап | Hot {slug} | Warm {slug} | Cold {slug} | Совокупная просадка |
|---|---|---|---|---|
| Трафик → Лид-магнит (% подписки) | 12% | 8% | 3% | средний |
| Лид-магнит → Tripwire (% оплаты TW) | 4% | 2% | — | **главный** |
| Tripwire → Core (% апсейла) | 25% | 15% | — | средний |
| Касания → Заявка | 3% | 1% | — | средний |
| Заявка → Сделка | 30% | 12% | — | при модуле встреч |

2. Применяет правило Голдрата: **главный bottleneck — один.** Это узкое место всей системы, а не «починить все одновременно».

3. Формулирует **сводное предложение** в `hypotheses.md`:

```markdown
## YYYY-MM-DD — сводный bottleneck воронки (фаза 3 финал)
Файл: hypotheses.md → Active
Гипотеза H{N}: [FUNNEL] главное узкое место — переход «Лид-магнит → Tripwire»,
конверсия 4% (hot) и 2% (warm). При среднем по нише 8-12%.
Причина (рабочая): TW позиционируется как мини-курс, а ЦА ждёт «попробовать»
формат — диагностику или PDF с быстрым выигрышем.
Тест: переписать описание TW под формат «5 вопросов и я говорю что у тебя
сломано», запуск 2 недели, метрика конв ≥ 6%.
Дедлайн: YYYY-MM-DD. Метрика провала: < 5% — откат, ищем другую механику.
```

4. Показывает клиенту:

> *«Все три сегмента распакованы. Главный bottleneck — переход с лид-магнита на tripwire: 4% у hot, 2% у warm при норме 8-12%. Это значит — мы притаскиваем людей и теряем их **до** того как они потратили первый рубль. Чинить надо это в первую очередь, остальное второстепенно. Гипотеза с тестом лежит в H{N} — принимаешь или хочешь обсудить?»*

Stop+wait — ждёт `/accept H{N}` или коррекцию.

---

### Шаг 6 — Финал фазы 3 + переход к фазе 4

После апрува Bottleneck-гипотезы — скилл закрывает фазу:

> *«Фаза 3 закрыта. По всем ТОП-сегментам прошита воронка, главный bottleneck зафиксирован, тесты в работе. Дальше — фаза 4, бренд: позиционирование, голос, визуальный слой. Сейчас Brand-агента в офисе ещё нет — ты можешь либо заполнить `brand/positioning.md`, `brand/voice.md` через существующие скиллы Дизайнера (`/brand-wheel`, `/positioning`, `/brand-voice`), либо подождать когда подключим отдельного помощника. Что выбираем?»*

**Важно:** Алекс **не запускает фазу 4 сам** — Brand-агент пока NOT WIRED в этом шаблоне. Алекс только информирует и передаёт.

---

## 5. Stop+wait чек-поинты — минимум 2 (плюс по числу сегментов)

| # | Когда | Что показывает | Что ждёт |
|---|---|---|---|
| 1..N | После каждого сегмента (внутри Шага 3) | Сводку по сегменту: канал, маршрут, касания, bottleneck, diff в Log | «принимаю / переделай / следующий или стоп» |
| Финал | После Шага 5 (сводный bottleneck) | Таблицу по всем сегментам + главный bottleneck + гипотеза-тест | `/accept H{N}` или коррекция |

Внутри `/funnel-build` — ещё 4 stop+wait по одному на каждый шаг диалога.

**Все чек-поинты — жёсткий stop+wait.** Без явного подтверждения клиента следующий шаг не запускается. Это закон голоса: владелец решает что попадает в `funnel/channels.md` и `welcome.md`, не Алекс.

## 6. Output contract

| Файл | Что туда попадает | Кто записывает |
|---|---|---|
| `hypotheses.md → Log` | diff-предложения по каждому сегменту: канал, LM, TW, маршрут, касания, метрики | **Алекс** (через предложение) |
| `hypotheses.md → Active` | до 5 активных гипотез с тегами `[CHANNEL]` / `[LM]` / `[FUNNEL]` / `[METRIC]` | **Алекс** + `/accept` от владельца |
| `funnel/channels.md` | Per-segment канальный контракт (этот сегмент — этот канал — этот CAC) | **Владелец** через `/accept H{N}` |
| `funnel/welcome.md` | Последовательность касаний, темы, форматы, CTA | **Владелец** через `/accept H{N}` |
| `funnel/scripts.md` | Скрипты + банк возражений | **Владелец** через `/accept H{N}` (только если активирован модуль встреч) |
| `metrics.md` | AUTO-секции по каналам если появились новые цифры | **Алекс** (только AUTO-зоны) |
| `audience/voc.md` | Возражения / отказы пойманные по ходу разговора | **Алекс** append-only |

**Закон голоса (важно):** Алекс **не перезаписывает** `funnel/channels.md`, `welcome.md`, `scripts.md` напрямую. Он формирует diff-предложение в `hypotheses.md`, владелец принимает командой `/accept H{N}` — только тогда файл вывода обновляется.

## 7. Если модуль встреч НЕ активирован

`funnel/scripts.md` не нужен — бизнес работает без созвонов (только тексты воронки и автомат-конверсии).

В этом случае:

- Шаг 3 «Touchpoints» внутри `/funnel-build` **не предлагает** скрипты созвона
- Возражения копятся в `audience/voc.md` с тегом `#objection`
- Эти возражения используются для оптимизации `welcome.md` (текстовые касания закрывают возражения которые в живой беседе закрылись бы продажником)
- diff в Log идёт только по `channels.md` + `welcome.md`

Если в процессе разговора с клиентом всплывает «надо подключить созвоны» — Алекс предлагает запустить `/marketer-enable-meetings`, **но не делает это сам**.

## 9. Red flags — когда «СТОП — переделать»

Скилл сам отслеживает:

- **Pre-flight не пройден** → не запускать диалог, отправить на нужный скилл
- **Ни одного канала из Core Four не подтверждён данными** (всё «гипотеза, я думаю») → принудительно сделать ось гипотезой `[CHANNEL]` с дедлайном и метрикой проверки
- **Канал выбран без обоснования из dossier** («у нас Instagram потому что у всех Instagram») → переделать Шаг 1 в `/funnel-build`
- **Один маршрут на все сегменты** → красный флаг унификации, разделить per-segment
- **Скрипты созвона предлагаются без модуля встреч** → удалить, оставить только текстовые касания
- **Возражения в скриптах — «из головы»** (нет ссылки на `voc.md` или цитаты клиентов конкурентов) → переделать
- **Bottleneck-анализ без цифр** («чувствую что течёт здесь») → требовать минимум одну метрику или явную пометку `[нет данных, гипотеза N%]`

- [ ] Для каждого ТОП-сегмента в работе — пройдены все 4 шага `/funnel-build` (channel → route → touchpoints → conversions)
- [ ] Stop+wait после каждого сегмента пройден, есть отметка в `hypotheses.md`
- [ ] В `hypotheses.md → Log` лежит минимум 1 diff-предложение по сегменту (канал + касания)
- [ ] Сводный bottleneck-анализ сделан, главный bottleneck зафиксирован отдельной активной гипотезой
- [ ] Владелец сделал `/accept H{N}` хотя бы по одному предложению (или явно сказал «всё в backlog, ничего сейчас не принимаю»)
- [ ] `funnel/channels.md` и `welcome.md` отражают принятые гипотезы
- [ ] При активном модуле встреч — `funnel/scripts.md` тоже обновлён
- [ ] Возражения / отказы пойманные по ходу — в `audience/voc.md` append-only
- [ ] Финальное сообщение клиенту: фаза 3 закрыта + информация о фазе 4 (Brand NOT WIRED)

---

## 11. Принципы скилла (короткий чек-лист)

- **Один сегмент за раз** — не распаковывать hot/warm/cold параллельно
- **Channel-First** — канал primary выбирается из dossier, не из «все так делают»
- **Bottleneck — один** — Голдрат: главное узкое место одно, чинить его, не везде
- **Per-segment маршрут** — у каждого сегмента свой LM, TW, последовательность касаний
- **Возражения снизу** — из `voc.md` и реальных цитат, не из головы
- **Скрипты только при модуле встреч** — без активации `/marketer-enable-meetings` `scripts.md` не трогаем
- **Закон голоса** — Алекс не перезаписывает `funnel/*.md`, только diff в Log → `/accept` владельца
- **Stop+wait жёстко** — после каждого сегмента и на финальном bottleneck
- **Em-dash `—`, не `--`** в любых текстах файлов

## 12. Память скилла (обязательно)

**Перед задачей:** `grep` по `office/agents/marketer/failures.md` на «воронка», «канал», «bottleneck», «scripts» — не повторяй прошлых ошибок. Прочитай `memory.md` секцию «Context» — что помнить из прошлых распаковок воронок этого клиента.

**После задачи:** append в `memory.md`:
- **Decisions** — какие каналы выбраны primary для каких сегментов, какой главный bottleneck
- **Patterns** — что заметил в клиенте (как принимает diff, на чём срывается, какие предложения отвергает)
- **Context** — что помнить в следующей итерации воронки этого клиента

Если bottleneck диагностирован неверно / `/accept` пришёл без правок но через неделю канал не дал результата / маршрут переделывался — обязательно в `failures.md`: `YYYY-MM-DD → что предложил → что оказалось → правило на будущее`.

**Append-only.**
