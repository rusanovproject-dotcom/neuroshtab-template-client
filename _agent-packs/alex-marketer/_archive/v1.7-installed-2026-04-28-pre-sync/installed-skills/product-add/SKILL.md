# /product-add — Добавление второго Core продукта

> Когда первый Core уже распакован (через `/product-build`), а владельцу нужен **второй или N-й Core продукт** в лестнице — этот скилл проводит через протокол `add-product.md`. Создаёт новый файл, очищает поля по шаблону, выстраивает Bridges между Core'ами, обновляет ladder.
>
> Опционально может **запустить полный пайплайн 01-06** для нового продукта — если владелец хочет распаковать его так же глубоко как первый.

---

## 1. Зачем

- Добавление второго / третьего Core когда первый уже собран
- Не каждому Core нужен полный 6-протокольный пайплайн — иногда достаточно копировать структуру и переписать ключевые поля
- Главное — **Bridges** между Core'ами: как клиент переходит с одного на другой, как они дополняют друг друга
- Output: новый Core в ladder, ссылка из основного `core-offer.md`, Bridges зафиксированы

---

## 2. Pre-flight checks

| Что | Где | Если нет |
|---|---|---|
| Первый Core распакован | `product/core-offer.md` заполнен (минимум VE + Stack + цена) | Стоп. Скажи: *«Сначала распакуй первый Core через `/unpack-product` — без него мост строить некуда».* |
| `_protocols/add-product.md` | `product/_protocols/add-product.md` | Если нет — lite-режим (см. секцию 5) |
| `product/ladder.md` | существует | Если нет — создаём в процессе |
| Хотя бы один сегмент в NORTH-STAR | `audience/segments/NORTH-STAR.md` | Стоп. Без сегмента не понятно для кого второй Core |

---

## 3. Алгоритм (на основе `_protocols/add-product.md`)

### Шаг 1 — Спросить базовые параметры нового продукта

Один вопрос за раз, живо:

1. *«Как называется новый продукт коротко? Slug — что-то типа `core-v2-premium` или `audit-90-days`. На английском, через дефис.»*
2. *«Какой tier? Entry (низкий вход), Core (основной), Premium (high-ticket)? Цены сам подставишь под свою экономику.»*
3. *«Pricing model: one-time (разовая оплата), subscription (подписка), retainer (ежемесячно), milestone (поэтапно)?»*
4. *«Под какой сегмент? hot/warm/cold или конкретный slug из NORTH-STAR?»*
5. *«Это самостоятельный продукт или продолжение первого Core (апселл)?»*

**Stop+wait:** *«Понял: {slug} {tier} {pricing} под {segment}, {standalone/upsell от первого Core}. Поехали?»*

### Шаг 2 — Создать файл нового продукта

**Структура (берём из `_protocols/add-product.md`):**

Создаём `product/cores/{slug}.md` (если папки нет — создаём `product/cores/`).

Альтернатива: дополнить основной `product/core-offer.md` секцией `## Core 2: {slug}` — выбор зависит от сложности. Если новый Core принципиально другой — отдельный файл. Если это вариация — секция в общем файле.

**Frontmatter нового файла:**

```yaml
---
slug: {slug}
tier: {entry/core/premium}
pricing_model: {one-time/subscription/retainer/milestone}
segment: {slug сегмента}
relation: {standalone/upsell-from-core-1/downsell-from-core-1}
created: YYYY-MM-DD
status: draft
---
```

### Шаг 3 — Очистить поля по шаблону

Копируем структуру первого `core-offer.md`, очищаем содержимое, оставляем заголовки секций:

- Дельта (where → where) — пусто, заполнять
- Аватар — пусто
- Value Equation 4 переменные — пусто
- Named Mechanism — пусто (или наследуем от первого Core если apsell)
- Stack — пусто
- Цена + RR + Downsell — пусто
- Оффер 1/4/8-10 — пусто
- Bridges — **тут будет связь с первым Core**

**Stop+wait:** *«Шаблон создан, поля очищены. Идём дальше — Bridges. Это самая важная часть для второго продукта.»*

### Шаг 4 — Заполнить Bridges (12/13)

Это **главная ценность `/product-add`** — без Bridges два Core'а не работают как лестница.

**12 — Bridge между сегментами:**
- Если первый Core под hot, второй под warm — как мостить?
- Какие триггеры переключения сегмента?
- Где в воронке клиент делает выбор между Core 1 и Core 2?

**13 — Bridge между Core'ами (как клиент переходит):**
- **Sequential** — сначала Core 1, потом Core 2 (классическая лестница). Триггер перехода — какой результат должен быть достигнут.
- **Parallel** — Core 1 и Core 2 для разных нужд одного клиента. Триггер — на какой задаче берёт что.
- **Either-or** — Core 1 или Core 2 — клиент выбирает одно. Триггер — диагностика на этапе оффера.

**Что записываем в Bridges:**
- Тип связи (sequential / parallel / either-or)
- Триггер перехода (какой результат / какая задача / какая диагностика)
- Сценарий: «клиент закончил Core 1 → видит результат [X] → предлагаем Core 2 потому что [Y]»
- Цена перехода (есть ли скидка для тех кто уже взял Core 1)

**Stop+wait:** *«Bridges собрал. Тип связи — {sequential/parallel/either-or}. Триггер — [X]. Согласен?»*

### Шаг 5 — Обновить ladder.md

Добавляем новый Core в `product/ladder.md`:

```
LM (lead-magnet)
  ↓
TW (tripwire)
  ↓
Core 1: {основной slug}  ←— уже есть
  ↓ Bridge {тип, триггер}
Core 2: {новый slug}     ←— добавляем
  ↓
PM (premium maximizer)
```

Diff в `hypotheses.md → Log`. Владелец принимает через `/accept`.

### Шаг 6 — Запись в `product/iterations.log`

Append в `product/iterations.log` (если файла нет — создаём):

```
## YYYY-MM-DD — добавлен Core: {slug}
- tier: {entry/core/premium}
- pricing: {model}
- relation to Core 1: {sequential/parallel/either-or}
- bridge trigger: {что}
- статус: draft (нужно распаковать через /product-build или заполнить вручную)
```

### Шаг 7 — Опциональный запуск полного пайплайна

Спроси владельца:

> *«Карточка продукта создана, Bridges зафиксированы. Дальше два пути:*
> *— Полный пайплайн 01-06 для этого продукта — пройдёшь все 6 протоколов как для первого Core. 60-90 минут, на выходе полный Grand Slam.*
> *— Заполнить вручную / lite-распаковка — если продукт похож на первый, можем перенести 70% полей и точечно докрутить.*
> *Что выбираешь?»*

- Если **полный пайплайн** → запусти `/product-build` с активным сегментом и пометкой `mode: additional-core, parent: {первый slug}`. `/product-build` пройдёт 6 протоколов для нового продукта.
- Если **lite** → проведи через 7-10 вопросов lite-режима `/product-build`.
- Если **вручную** → отдай владельцу шаблон с очищенными полями, объясни что заполнить.

**Stop+wait:** *«ОК, идём через [полный/lite/вручную]. Начинаю.»*

## 4. Output

| Файл | Что туда |
|---|---|
| `product/cores/{slug}.md` (или секция в `core-offer.md`) | Новый Core — frontmatter + очищенные секции |
| `product/ladder.md` | Обновлённая лестница с новым Core |
| `product/iterations.log` | Запись о добавлении |
| `hypotheses.md → Log` | Diff для ladder.md и core-offer.md (ссылка на новый Core) |
| `hypotheses.md → Active` | Гипотеза `[PRODUCT]` про новый Core |

**Если запускали полный пайплайн** → дополнительно output от `/product-build`.

**Закон голоса:** Алекс не пишет в `cores/{slug}.md` напрямую как в финал — создаёт draft, владелец принимает через `/accept` или докручивает руками.

## 5. Lite-режим (если `add-product.md` нет)

Если протокола `_protocols/add-product.md` нет — упрощённая версия:

1. Спроси slug + tier + pricing-model + segment + relation (5 вопросов)
2. Создай draft файла `product/cores/{slug}.md` с базовым frontmatter и пустыми секциями (как в первом core-offer)
3. Заполни Bridges через 3 вопроса:
   - Тип связи с первым Core (sequential / parallel / either-or)
   - Триггер перехода
   - Сценарий одной фразой
4. Обнови `ladder.md`
5. Спроси про распаковку — `/product-build` lite или вручную

Stop+wait — после шагов 1, 3, 5.

Пометка в шапке файла: *«lite — добавлено без полного протокола add-product. Для полной распаковки — клонировать project-template/product/_protocols/».*

## 6. Anti-patterns

- **Не добавляй второй Core если первый недопакован.** Если в `core-offer.md` пусто — стоп, сначала первый.
- **Не копируй Value Equation 1-к-1 из первого Core.** Это другой продукт, другой Dream Outcome, другой Likelihood. Только структура.
- **Не пропускай Bridges.** Без них — это два разных продукта, а не лестница. Bridges — главное.
- **Не выдумывай triger перехода.** Триггер должен быть из реального сценария клиента, а не «логично что после Core 1 нужен Core 2».
- **Не предлагай 3+ Core'а сразу.** Один за раз. Если владелец хочет ещё — отдельный заход `/product-add`.
- **Не апсель без причины.** Если первый Core полностью закрывает Dream Outcome — может быть второй Core не нужен, а нужен только PM (premium maximizer) или подписка на retainer.
