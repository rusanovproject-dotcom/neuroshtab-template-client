# Процедуры свёртки, усечения, навёрстывания

Однозначные алгоритмы исполнителей WEEKLY и MONTHLY. Свёртки используют их обе — потому процедуры здесь, не размазаны по шаблонам.

---

## Процедура свёртки WEEKLY

**Исполнитель:** WEEKLY. **Триггер:** вс вечер ИЛИ `state.last_weekly < current_week`.

```
1. Читает metrics-YYYY.jsonl за 7 дат недели + итоги daily недели (секция «Итог дня» каждого дня).
2. Считает на лету: спарклайн day_score, средний day_score, Δ к прошлой неделе,
   % задач (Σ tasks_done / Σ tasks_planned), стрик+лучший, дни rest/freeze, coverage(строк/7).
   → Если суммарно >10K токенов: запустить УСЕЧЕНИЕ (см. ниже) ПЕРЕД сборкой отчёта.
3. Собирает отчёт-комбо из weekly/TEMPLATE.md:
   опора (что держало) · траектория смысла (по «Итогам» и meaning) · что исчезло (тема была — пропала).
4. Сверяет с goals.md (фокусы недели): что из week_focuses сдвинулось, что нет.
5. Обновляет patterns/: новые точки в гипотезы, проверка порогов (hypothesis→confirmed→core). Сюда же — корреляции `energy`↔`sleep_h`/`workout` за неделю (читатель полей sleep_h/workout: кандидаты в `patterns/energy`, помечать «корреляция, не причина»). **Читатели новых полей (2026-05-24):**
   - **дни высокого `focus`** → паттерн фокуса в `patterns/productivity` («собран, когда…»); дни низкого `charge` → что его роняет (сон/стресс/энергия — видно по компонентам).
   - **Калибровка `sleep_h`↔`rested`:** устойчивая связка (сколько часов даёт ощущение «выспался») → личная норма сна. Норма стабилизировалась → пишу в `core/profile.md` операционку («норма сна ~X ч») — это персональный пик `sleep_score` (metrics-schema, вместо дефолтных 8ч). Расхождение «спал много / rested низкий» устойчивое → гипотеза в `patterns/energy` (качество сна, не часы).
6. Пишет weekly/YYYY-WXX.md (отчёт + cache-блок).
7. Обновляет state.md: last_weekly = current_week.
8. Если свёрнута ПОСЛЕДНЯЯ неделя месяца И state.last_monthly < этот месяц → запустить MONTHLY-свёртку следующим шагом (страховка без планировщика: не ждать 1-го числа, размыкает промоушн в ядро).
```

Промоушн в ядро на этом шаге НЕ делается — он только в MONTHLY.

**Закрытие floor (тот же день, даже при минимуме).** Разбор, начатый сегодня, сворачивается сегодня. Даже если человек вымотан и ceiling (фокусы + глубокий разбор) переносится — floor исполняется всё равно: шаги 3 (образ + опора), 6 (пишет `weekly/YYYY-WXX.md`) и 7 (`last_weekly = current_week`) обязательны. `last_weekly` ставится при ЛЮБОМ закрытии — иначе неделя всплывёт заново как несвёрнутая (ложный повтор WEEKLY). Перенесённый остаток → строка в `open-loops` (кто грузит: MORNING), чтобы доразбор поднялся утром по обещанию. Открытый и молча брошенный недельный чек — нарушение процедуры (ведение floor/ceiling — `skills/review.md`).

### Вход свёртки дампом/транскриптом (способы B/C) — дедуп и конфликт

Когда WEEKLY входит не ведомой сессией, а свободным дампом или транскриптом (роутер входа — `router.md`):
- **Дедуп против daily:** факты дампа, уже записанные в daily недели, в нарратив свёртки НЕ задваиваются. Сверка по датам + «Итогам дня».
- **metrics неприкосновенны:** дамп/транскрипт НЕ создаёт и НЕ переписывает строки `metrics-YYYY.jsonl` задним числом. Числа пишет только дневной контур (evening / morning-догон / capture). Дамп идёт в НАРРАТИВ свёртки, не в jsonl.
- **Конфликт текст-vs-цифры:** «верим тексту как его правде, цифра — повод прочитать» (SPEC §3). Расхождение мягко отражается в блоке «что я увидел», jsonl не трогается.
- **Усечение >10K** применяется и ко входному транскрипту (способ C): вход свёртки = daily + дамп/транскрипт, режется по тому же приоритету (daily-тексты до «Итога», metrics и траектория смысла — никогда).
- **Идемпотентность:** повторный запуск той же недели не плодит второй `weekly/YYYY-WXX.md` — обновляет блоком `## YYYY-MM-DD пересмотр`, `last_weekly` не двигается назад.

## Процедура свёртки MONTHLY + промоушн

**Исполнитель:** MONTHLY. **Триггер:** 1-е число ИЛИ месяц не свёрнут.

```
1. Читает ТОЛЬКО 4-5 weekly месяца + их cache-блоки. daily НЕ открывает.
2. Строит тренд по неделям (таблица из monthly/TEMPLATE.md): линия day_score, стрика, % задач.
3. ПРОМОУШН:
   - confirmed/core-паттерн из patterns/  →  стратегия в core/what-works.md (≤25, слабейшее в _archive)
   - устойчивый факт (пики энергии и т.п.) →  core/profile.md операционка
4. Пишет monthly/YYYY-MM.md.
5. АРХИВ: daily-файлы свёрнутого месяца переезжают в _archive/YYYY-MM/.
   Свёртка их уже впитала, в daily/ они больше не нужны (за 18 мес иначе ~550 файлов).
   metrics-YYYY.jsonl остаётся на месте — это источник правды, не сырьё.
6. ОБЯЗАТЕЛЬНО: state.md ← last_monthly + что промоутнул.
   Промоушн без записи в state = НЕ ВЫПОЛНЕН (повторится, задвоит ядро).
```

---

**Колесо (8 сфер) в MONTHLY.** Месячный разбор (`review` месячная оптика) ПЕРВЫМ шагом собирает Колесо живьём (цифры + контекст) и пишет снимок в блок «Колесо» `monthly/YYYY-MM.md`. Свёртка сравнивает сферы с прошлым `monthly/` (первый месяц — с `state.wheel_baseline`), считает сдвиг (Δ), проседающую 2 мес подряд помечает кандидатом в `goal-setting`. **Цифры сфер НЕ идут в `metrics-YYYY.jsonl`/Заряд** — это месячный срез, не дневная метрика. Floor: дал не все 8 → что есть.

## Процедура УСЕЧЕНИЯ (масштаб 6-18 мес — иначе WEEKLY распухнет молча)

**Когда:** на шаге 2 свёртки WEEKLY входные данные суммарно **> 10 000 токенов**.

Алгоритм (по приоритету, режем сверху вниз пока не уложились в 10K):
```
РЕЖЕТСЯ:
  1. daily-тексты усекаются до строки «Итог дня».
     Секции Тело / День / Эмоция / задачи — отбрасываются из входа свёртки.
     (Сами файлы daily НЕ трогаются — усечение только на входе в свёртку.)

НЕ РЕЖЕТСЯ НИКОГДА:
  - metrics-YYYY.jsonl берётся ЦЕЛИКОМ (числа дёшевы, это источник правды).
  - траектория смысла (линия meaning + «Итоги» про смысл) — не режется,
    даже если daily усечены до «Итога»: смысл собирается именно из «Итогов».
```

Порог: **10K токенов**. Что режется: **daily-тексты → до строки «Итог дня»**. Что не режется: **metrics целиком + траектория смысла**.

**Если после усечения daily вход всё ещё >10K** — это сигнал, не баг (неделя необычно плотная или несколько недель слиплись при навёрстывании). Эскалируй владельцу: «неделя не влезает даже после усечения, как сворачиваем». metrics и траекторию смысла НЕ режем — лучше спросить, чем потерять смысл.

## Пересчёт стрика (детерминированный, по календарным датам)

**Исполнитель:** режим, который ЗАКРЫВАЕТ чек — **EVENING** (штатно) и **MORNING при догоне вчерашнего чека** (`morning-check` пишет `check_done` за вчера → обязан тут же пересчитать кеш). **CAPTURE стрик НЕ трогает** — захват чек не закрывает (`capture.md`), значит и кеш не пересчитывает. **Не** на каждый входящий — только в момент закрытия чека.

Стрик = непрерывность СДЕЛАННОГО ЧЕКА, не выполнения задач. Считаем НЕ по числу строк, а по календарю. Берём строки `metrics-YYYY.jsonl`, сортируем по `date`, идём от свежей даты НАЗАД, по одной календарной дате за шаг.

⚠️ **Порядок проверки внутри даты: сначала `day_type`, потом `check_done`.** Freeze-строка имеет `check_done:none` (см. схему) — если проверять `check_done` первым, freeze ошибочно порвёт цепь. Сначала смотрим `day_type` (freeze/rest), и только потом `check_done`.

```
1. day_type: freeze за дату                        → стрик СОХРАНЯЕТСЯ (пауза: дата не рвёт цепь, но и не +1 — перешагиваем)
2. day_type: rest                                  → +1 (отдых — выполненная задача дня)
3. check_done: full ИЛИ min                        → +1
4. строка есть, но check_done: none И не freeze    → стрик РВЁТСЯ в 0 (человек активно отметил пустой день)
5. пропущенная ПРОШЛАЯ дата (нет строки И не freeze) → стрик РВЁТСЯ в 0 на этой точке, пересчёт назад останавливается
- GRACE на сегодня: отсутствие строки за СЕГОДНЯ стрик НЕ рвёт — пропускаем сегодня, считаем от вчера. Рвёт только пропущенный прошлый день.
```

Почему по датам, а не по строкам: при заморозках и пропусках число строк ≠ длине цепи (freeze-строки есть, строк за пропуск нет). Счёт по строкам врёт задним числом — счёт по календарю детерминирован.

**Единый момент пересчёта-записи = каждый чек.** На закрытии чека стрик пересчитывается из jsonl по датам, кеш `стрик`/`лучший` в таблице `commitments.md` сверяется с пересчётом. Расходятся — кеш переписать (истина — пересчёт по датам, не кеш). Другого момента синхронизации нет — кеш не живёт своей жизнью. Милстоуны 7 / 30 / 100 — повод отметить: если после пересчёта кеш ВПЕРВЫЕ достиг 7/30/100 → поднять `celebrate` прямо здесь, КТО БЫ ни закрыл чек (evening / morning-догон / rest / freeze-сохранение). Это процедурный шаг, не привязан к одному навыку — иначе милстоун, добитый утренним догоном, проглатывается молча. Слом → любопытство, не приговор.

## Процедура НАВЁРСТЫВАНИЯ (несвёрнуто >1 периода)

**Когда:** между last_weekly и current_week больше одной недели (пропущены свёртки).

```
Сворачиваем ПО ОДНОМУ ПО ПОРЯДКУ: сначала W21, затем W22, затем W23…
Не прыгаем и не сворачиваем «оптом» — каждый период отдельным проходом и отдельным файлом.

При навёрстывании daily грузим ТОЛЬКО метрики + «Итоги дня» (не полный текст) —
это снимает нагрузку на догон и совпадает с правилом усечения.

После каждого периода — обновить state.last_weekly, затем следующий.
Аналогично для MONTHLY: месяц за месяцем по порядку.
```
