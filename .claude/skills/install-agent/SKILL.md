# install-agent — универсальный установщик помощников

## Зачем

Базовый офис пользователя минимальный: Director + Стратег + Архитектор. Остальные помощники (Дизайнер, Копирайтер, Продажник, Tech Lead и т.д.) лежат готовыми паками в `_agent-packs/`. Пользователь ставит их когда готов — одной командой, без ручного редактирования конфигов.

Этот скилл — универсальный установщик. Читает манифест пака (`_agent-packs/<name>/install.md`), применяет изменения, оповещает пользователя живым языком.

## Когда запускать

**Явно:**
- `/install-agent <name>` — прямая команда с именем пака

**Неявно (из фразы пользователя):**
- *"установи дизайнера"* / *"подключи копирайтера"* / *"добавь в команду продажника"*
- *"хочу дизайнера"* / *"нужен помощник по <тема>"* — если есть пак под эту тему, предложи установку
- *"какие помощники можно подключить?"* — покажи список доступных паков

**Не запускать:**
- Агент уже установлен (проверь `office/AGENTS.md`) — сказать пользователю «уже в команде», предложить переустановку если пользователь хочет свежие файлы
- Пака не существует в `_agent-packs/` — показать список доступных, честно сказать что такого пока нет

## Алгоритм

### Шаг 1. Определить имя пака

- Если вызвано как `/install-agent <name>` — берём `<name>`.
- Если из фразы — маппинг ключевых слов на имя пака (только реально существующие паки в `_agent-packs/`):
  - «маркетолог ЦА», «JTBD», «jobs to be done», «распакуй аудиторию», «распакуй ЦА», «синтетический касдев», «алекс» → `alex-marketer`
  - «маркетинг рынок», «оффер», «воронка», «позиционирование», «сегменты», «продуктовая лестница» → `marketer`
  - «дизайнер», «designer», «визуал», «лендинг», «брендбук», «презентация», «обложка» → `designer`
  - «копирайтер», «copywriter», «тексты», «пост», «лонгрид», «рассылка» → `copywriter`
  - «коуч», «нейрокоуч», «neuro-coach» → `neuro-coach`
  - «рита», «хранительница», «аудит офиса», «наведи порядок», «проверь офис», «office-cleaner» → `office-cleaner`
  - «демиург», «архитектор команды», «собери агента», «новый помощник» → `demiurg`

Если ключевые слова неоднозначны (например, просто «маркетолог» — Алекс по JTBD или Маркетолог по офферу/рынку?) — спроси живым языком: *«Тебе помощник по распаковке аудитории через JTBD (Алекс) или по офферу / воронке / рынку (Маркетолог)?»*

**Несуществующий пак — не выдумывай.** Если фраза указывает на роль, которой пока нет в `_agent-packs/` (продажник / sales, tech lead / разработчик и т.п.) — НЕ подставляй имя пака наугад. Честный fallback: *«Готового помощника по [роль] пока нет. Есть: [список из «Доступные паки сейчас»]. Что-то из этого подключить?»* (дальше — Шаг 2c).

### Шаг 2. Проверить существование пака

**2a. Проверь локальный пак:** наличие `_agent-packs/<name>/install.md`.

**2b. Если локально нет — скачай с GitHub автоматически:**

- `<AGENT_PACKS_REPO>` в формате `<owner>/<repo>` (подставляется из .env.example → AGENT_PACKS_REPO_URL; по умолчанию — репозиторий владельца офиса)
- **Репозиторий паков:** `https://github.com/<AGENT_PACKS_REPO>`
- **Raw base URL:** `https://raw.githubusercontent.com/<AGENT_PACKS_REPO>/main`
- **API для списка файлов:** `https://api.github.com/repos/<AGENT_PACKS_REPO>/contents/_agent-packs/<name>`

Алгоритм:
1. Пользователю живым языком: *«Секунду, подтяну с репозитория.»*
2. Создай `_agent-packs/<name>/`
3. Через GitHub API получи рекурсивно список всех файлов пака
4. Каждый файл скачай: `curl -sL <raw-url> -o <local-path>` с сохранением структуры папок
5. Проверь что `_agent-packs/<name>/install.md` теперь локально
6. Если скачивание не удалось (нет сети / репо недоступно / пака нет в репо): *«Не смог подтянуть пак <name>. Проверь интернет или напиши владельцу офиса.»*

**2c. Если пака нет ни локально ни на GitHub:**
1. Через API получи список доступных паков: `https://api.github.com/repos/<AGENT_PACKS_REPO>/contents/_agent-packs`
2. Пользователю: *«У меня пока нет пака для <name>. Есть готовые: [список]. Что ставим?»*
3. Если список пустой: *«Пока готовых паков нет. Владелец офиса добавит — я подключу.»*

### Шаг 3. Проверить что агент ещё не установлен

Проверь `office/agents/<name>/` — существует ли папка.

**Если агент уже установлен:**
- Прочитай `office/AGENTS.md` на предмет строки про этого агента.
- Скажи пользователю: *«<Human name> уже в команде. Хочешь переустановить свежие файлы? Твои персональные правила и накопленная память я не трону — только обновлю ядро и знания.»*
- Если пользователь соглашается — переходи к шагу 4 в режиме переустановки (флаг `reinstall = true`, `preserve_if_exists` работает).
- Если отказывается — стоп.

### Шаг 4. Прочитать манифест

Открой `_agent-packs/<name>/install.md`. Извлеки:
- `agent_id`, `agent_name_human`, `agent_name_in_chat`, `short_role`
- `files:` — список src→dest + флаги `preserve_if_exists`
- `folders:` — папки которые надо создать если их нет
- Строки для `office/AGENTS.md`, корневого `CLAUDE.md`, `director/core.md`, `routing-patterns.md`
- `first_task` и блок «Message to client»
- `integration_model` — `inline` по умолчанию (агент грузится в офис-сессию через `@include`) ИЛИ `pointer` (агент — самодостаточная система со своим входом `CLAUDE.md`, запускается СВОЕЙ сессией в своей папке; см. «⚠️ Pointer-агенты» ниже). Если поля нет — считай `inline`.
- `Post-copy step` — команда, которую надо выполнить после копирования (напр. `bash .../bootstrap.sh`), если раздел есть в манифесте

**Если манифест битый** (нет agent_id / нет раздела files / нет сообщения пользователю):
> *«Пак <name> выглядит неполным — пропущены важные поля. Скажи владельцу офиса, он доправит. Пока устанавливать не буду — не хочу оставить офис в странном состоянии.»*

### Шаг 5. Скопировать файлы

Для каждой записи в `files:`:
- Создай папку назначения если её нет.
- Если `preserve_if_exists: true` и файл назначения существует — **пропусти**, не затирай.
- Иначе — скопируй из `_agent-packs/<name>/<src>` в `<dest>`.

Папки (`knowledge/`, `skills/`, `templates/`) копируй рекурсивно.

**Пак-«целая система»** (запись `src: ./` с `recursive: true`, напр. НейроКоуч) — копируй ВСЁ содержимое пака в `dest`, КРОМЕ файлов из `exclude` (как минимум сам `install.md`). Включая скрытые файлы (`.gitignore` агента — он нужен, вложенно игнорит личные данные относительно своей папки).

Для `folders:` — создай если не существует, оставь существующие README нетронутыми.

**Post-copy step.** Если в манифесте есть раздел «Post-copy step» с командой (напр. `bash office/agents/<name>/scripts/bootstrap.sh`) — выполни её после копирования. Это разворачивает личные файлы инстанса из шаблонов (идемпотентно).

### Шаг 6. Обновить `office/AGENTS.md`

Прочитай файл. Найди таблицу «Установленные агенты». Добавь строку из манифеста **после последней строки таблицы** (перед пустой строкой / разделителем).

Если в манифесте указана строка «Прямое обращение» — найди строку `**Прямое обращение** к агенту:` и допиши `@<agent_id>` если его там ещё нет.

Если в манифесте есть блок «Онбординг» для этого агента — вставь его в подходящее место (перед `---`).

Если агент уже был в таблице (защита от дубликатов) — **не дублируй строку**, оставь как есть.

### Шаг 7. Обновить корневой `CLAUDE.md`

**Layered include:** найди блок `@office/agents/.../core.md` (строки идут подряд). Добавь `@office/agents/<name>/core.md` после последней такой строки, если её ещё нет.

> ⚠️ **ЕСЛИ `integration_model: pointer` — НЕ добавляй `@include`.** Pointer-агент (напр. НейроКоуч) — самодостаточная система со своим входом, его НЕЛЬЗЯ грузить в каждую офис-сессию (подерётся за роутинг и память, к тому же у него точка входа `CLAUDE.md`, а не `core.md` — `@include core.md` будет битым). Манифест pointer-агента это прямо запрещает. Для него — ТОЛЬКО строка в «Типовые роли» ниже (осведомлённость), её формулировка в манифесте уже про «запускается своей сессией».

**Таблица «Типовые роли»:** найди таблицу в секции «Команда». Добавь строку из манифеста после последней существующей, если её ещё нет.

### Шаг 8. Обновить `office/agents/director/core.md`

**В секцию «Команда офиса»** (список «Типовые роли» — буллеты `- **Название** — описание`) — добавь строку из манифеста.

**В секцию «Роутинг»** — добавь строку из манифеста в «Базовое правило маршрутизации».

**Если в манифесте есть спецсекция (триггер-блок)** — вставь её перед секцией `## Output contract`. Проверь что такая секция ещё не вставлена (защита от дубликатов по заголовку).

### Шаг 9. Обновить `office/agents/director/knowledge/routing-patterns.md`

**В таблицу «Core-роутинг»** — добавь строки из манифеста, если их ещё нет.

**В секцию «Правила»** — добавь пункты из манифеста если они есть.

**В «Параллельный / последовательный режим»** — добавь связки из манифеста.

### Шаг 9.5. Спец-директивы манифеста (ОБЯЗАТЕЛЬНО, если есть)

Манифесты некоторых паков используют не только «добавь строку», но и директивы замены/вытеснения. Шаги 6-9 по умолчанию ДОБАВЛЯЮТ — но если в манифесте есть директивы ниже, обработай их как сказано, иначе получишь дубли.

- **`replace_line` / `replace_row` / `replace_if_present`** — это ЗАМЕНА, не добавление. Найди в целевом файле строку/табличный ряд по `match` / `match_contains`, и **замени** её на `with` / `with_rows` (а не дописывай рядом). Если совпадений несколько (match неуникален) — ориентируйся на инлайн-комментарии манифеста (какие именно ряды; обычно только табличные `| ... |`, не блокнот-примечания `> ...`). **Идемпотентность:** если целевые строки уже выглядят как результат замены (манифест обычно пишет маркер, напр. «если строка уже начинается с «рита» — пропусти») — НЕ заменяй повторно. Частый случай: `office-cleaner` — базовый шаблон уже маршрутит аудит на `/office-cleaner`, установка лишь обогащает ряды (триггеры по имени) — итог должен быть 3 ряда, не 6.

- **`supersedes: [X]`** (в Metadata) — этот пак вытесняет пак X. Если X **установлен** (есть `office/agents/X/` или его скиллы в `.claude/skills/`): перенеси конфликтующие скиллы X в `.claude/skills/_archive/<skill>-superseded/` (с пометкой deprecated) и сними его конфликтующие routing-строки, ИЛИ, если не уверен что безопасно — **явно предупреди пользователя**: «Ставлю Алекса как основного маркетолога — у базового Маркетолога пересекался скилл логирования встреч, отключаю его, чтобы не было двойного срабатывания. Ок?». Не оставляй два скилла с идентичными авто-триггерами активными.

- **`archive_skills` (секция Pre-install)** — ПЕРЕД копированием перенеси перечисленные устаревшие скиллы в указанный `dest` архива (`if_exists: true` → пропусти молча если нет), проставь `update_frontmatter` (status: deprecated, archived_on: сегодня). Сообщи в post-install одной фразой.

- **`settings_json` (секция Critical security)** — убедись что `.claude/settings.json` существует и содержит все `required_deny_rules`. Нет файла → создай из `template_path`. Есть, но deny неполный → допиши недостающие правила (deep-merge), не трогая `permissions.allow` и `hooks`. Без этого шага установку такого пака не завершать.

### Шаг 10. Оповестить пользователя

Покажи сообщение из блока «Message to client» манифеста. Живым языком, без IT-терминов.

Не произноси вслух:
- «скилл», «YAML», «install.md», «layered include», «манифест», «пак», «core.md», «routing-patterns»
- «агент», «artifact», «prompt», «слаш-команда»

Говори:
- «помощник», «подключил», «добавил в команду»
- «шаблон», «инструкция» (вместо промт)
- «живая страница» (вместо артефакт)

В конце сообщения — предложи первую задачу из `first_task.suggestion` одной фразой:
> *«Давай начнём с <first_task.suggestion>? <first_task.why> Скажи "<пример фразы>" и я запущу.»*

### Шаг 11. Самопроверка

**Обычный (inline) агент:**
- [ ] Папка `office/agents/<name>/` существует и содержит `core.md`
- [ ] В `office/AGENTS.md` есть строка про агента
- [ ] В корневом `CLAUDE.md` есть `@office/agents/<name>/core.md`
- [ ] В `director/core.md` есть упоминание агента в «Команде офиса»
- [ ] В `routing-patterns.md` есть триггер

**Pointer-агент (`integration_model: pointer`, напр. НейроКоуч):**
- [ ] Папка `office/agents/<name>/` существует и содержит точку входа `CLAUDE.md` (у pointer-агента вход — его `CLAUDE.md`, НЕ `core.md`)
- [ ] Выполнен `Post-copy step` (bootstrap развернул личные файлы)
- [ ] В `office/AGENTS.md` есть строка про агента
- [ ] В корневом `CLAUDE.md` `@include` НЕ добавлен (для pointer это правильно) — есть только строка в «Типовые роли»
- [ ] В `director/core.md` есть строка-указатель («открой отдельной сессией»)

**Пост-установочный health-check (обязательно):** прогони
```
bash _agent-packs/demiurg/scripts/validate-office.sh .
```
и убедись что нет новых BLOCKER/ERROR (агент зарегистрирован, ссылки целы). Если появились — доправь. Пользователю показывай только живой итог («Подключил, всё на месте»), не технический вывод.

Если что-то не прошло — скажи пользователю:
> *«Подключил, но заметил что <что именно> не обновилось. Проверь с владельцем офиса — возможно, файл был нестандартной структуры. Помощник в команде, работать должен.»*

## Guardrails

- **Не затирать `overrides.md` / `memory.md` / `failures.md`** если они уже есть. Это персональные данные пользователя и накопленная память агента — потеря = критично. Проверяй флаг `preserve_if_exists`.
- **Идемпотентность.** Повторная установка не должна дублировать строки в `AGENTS.md` / `CLAUDE.md`. Перед вставкой — проверка на наличие.
- **Битый манифест — стоп.** Если полей не хватает — не пытайся «додумать», скажи пользователю связаться с владельцем офиса.
- **Не спрашивай подтверждение на каждый файл.** Пользователь один раз сказал «установи» — ставь весь пак. Подтверждение нужно только при переустановке (будут затронуты ядро и знания).
- **Не упоминай в разговоре технические детали.** Пользователь не должен слышать: «сейчас пропатчу CLAUDE.md», «обновлю routing-patterns», «применю install.md». Говори: «подключаю помощника».
- **Закрытый пак.** Если в `install.md` есть `requires: [<other-agent>]` — проверь что те агенты уже установлены. Если нет — скажи пользователю: *«Для подключения <name> сначала нужен <required>. Хочешь, сначала его поставлю?»*
- **Эскалация при ошибке.** Если в процессе копирования что-то сломалось (нет прав, файл занят) — откати начатое где возможно и скажи пользователю что не получилось и почему.

## Доступные паки сейчас

Источник правды — содержимое `_agent-packs/` (исключая `*-backup-*` и отключённые `install.md.DISABLED`). На момент написания — 7 паков:

- **alex-marketer** — Алекс Маркетолог. Распаковка ЦА через JTBD (13 шагов на сегмент) + опционально синтетические касдевы. Основной маркетолог (`supersedes: marketer`).
- **marketer** — Маркетолог. Оффер, воронка, позиционирование, сегменты, гипотезы, клиентская база.
- **copywriter** — Копирайтер. Посты, лендинги, лонгриды, рассылки в голосе автора (сборка голоса через `/voice-dna`).
- **designer** — Дизайнер. Brand Book + лендинги / обложки / презентации (Claude Code + 4 MCP-библиотеки) + деплой на Vercel.
- **neuro-coach** — НейроКоуч. Фоновый коуч (энергия, рефлексия, фокус). Самодостаточная система (`integration_model: pointer` — запускается своей сессией).
- **office-cleaner** — Рита, Хранительница офиса. Ревизия структуры, памяти, роутинга. 3 режима: scan / tidy / deep.
- **demiurg** — Архитектор команды. Сборка новых помощников вокруг знаний (`/build`), реорганизация офиса.

Когда появятся новые — они просто лягут в `_agent-packs/` и станут доступны автоматически.
